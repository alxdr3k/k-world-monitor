  #### (3.3) quota_reservation atomic cap (GPT T-7)

  ```sql
  CREATE TABLE quota_reservation (
    reservation_id     TEXT PRIMARY KEY,      -- res_<ULID>
    scope_type         TEXT NOT NULL CHECK (scope_type IN ('turn','round','session','daily','weekly','tier0','backfill')),
    scope_id           TEXT NOT NULL,         -- turn_<ULID> / rnd_<ULID> / sess_<ULID> / YYYY-MM-DD / etc
    vendor             TEXT,                  -- openai / anthropic / google / NULL for cross-vendor
    tier               INTEGER,               -- 0~3 or NULL
    estimated_cost_usd REAL NOT NULL,
    actual_cost_usd    REAL,
    status             TEXT NOT NULL CHECK (status IN ('reserved','committed','released','expired')),
    created_at         TEXT NOT NULL,
    expires_at         TEXT NOT NULL          -- reserved 상태 자동 expire (10min default)
  );
  CREATE INDEX quota_reservation_scope_idx ON quota_reservation(scope_type, scope_id, status);
  ```

  atomic pattern (`BEGIN IMMEDIATE` 안에서 read total + insert):

  ```
  BEGIN IMMEDIATE
    -- null-safe single-value per row (Codex round 12 finding):
    -- actual 우선, NULL 이면 estimated. reserved + committed 모두 single cost.
    read sum(COALESCE(actual_cost_usd, estimated_cost_usd)) for scope
    if would exceed cap -> reject
    insert reservation status='reserved'
  COMMIT
  LLM call
  BEGIN IMMEDIATE
    update reservation actual_cost / status='committed'
  COMMIT
  ```

  backfill 은 별도 bucket — daily cap 와 분리되지만 global emergency hard
  cap 은 공유.

## Discussion

위 frontmatter `proposed_answer` 에 6 라우팅 경로 + flag priority + schema
신규 + CLI 명령 표준 + AI intent classifier 정리.

핵심 원칙:
- **DB 에는 항상 명시적** — 모든 질문은 session_id / round_id / turn_id 로 기록
- **CLI UX 는 active context 제공** — operator 매번 ID 입력 회피

- **classifier 는 보조** — auto routing 사고 방지

### Boundary with Q-050 (research orchestration layer)

Q-050 = research orchestration layer (ExplorationRound + RoundSynthesis +
SearchOrchestrator + RoundContextPack). Q-051 = 그 layer 위의 CLI UX +
ResearchTurn (Turn = round 안의 개별 user question/answer).

ResearchTurn 이 ExplorationRound 의 child 객체. 한 round 안에 N turns:

```
ResearchSession
  └─ ExplorationRound (Q-050)
       └─ ResearchTurn (Q-051)
            └─ RoundSearchRun (Q-050)
                 └─ RoundCandidateUrl (Q-050)
```

즉 turn 은 user-facing 단위, round 는 orchestration 단위, session 은
조사 주제 단위.

### 운영자 명시 vs AI classifier 분리

- **명시 routing** = operator 의 "이 질문은 어디로" 명시 의도. 우선순위 최고.
- **active context** = 직전 CLI session 의 state. 편의용.
- **classifier** = LLM 의 "이 질문은 topic shift" 추측. 보조용. 절대 단독
  override X. sensitive route (fork / branch) 는 사용자 확인 의무.

### Decision lock 본문 (Q-I / Q-J / Q-K)

#### Q-I — 발행 가능 단계 마킹

ResearchSession schema 에 `ready_for_thesis_compose BOOLEAN NOT NULL DEFAULT 0`
컬럼 추가. 운영자가 `/ops/sessions/:id/mark-ready` 명시 trigger 또는
CLI `pipeline research mark-ready <session_id>` 호출 시 true 전환.
true 전환 후 RoundSynthesis → Thesis compose entry condition.

automation 없음 (hypothesis_delta 비어 있어도 자동 transition X / confidence
도달해도 자동 X). 운영자 명시만.

#### Q-J — fork_session enum

ResearchSession schema 의 `forked_from_session_id` 컬럼에 추가:
`fork_kind TEXT`. fork 가 아닌 session 은 둘 다 NULL, fork session 은
둘 다 NOT NULL 의무 — table-level conditional CHECK 으로 enforce:

```sql
CHECK (
  (forked_from_session_id IS NULL     AND fork_kind IS NULL)
  OR
  (forked_from_session_id IS NOT NULL AND fork_kind IN ('derivative', 'side_exploration'))
)
```

값 의미:
- `derivative`: 별도 publication 후보 — 결국 별도 발행 흐름. 라운드
  evolution 후 ready_for_thesis_compose 자동 후보.
- `side_exploration`: 임시 idea 탐색, 발행 의도 없음. 자동 publication
  trigger 안 됨.

fork 시점에 operator 명시 의무. CLI: `pipeline ask --fork-session
--from-session sess_... --fork-kind {derivative|side_exploration} --title
"<title>" "<message>"` (위 line 256-262 의 CLI standard 와 일관) 또는
P1+ web `/ops/ask` UI dropdown (RESEARCH-1A.1 슬라이스).

#### Q-K — exploratory → directed transition

research_session.mode 가 `exploratory` 인 session 이 `directed` 로
transition 시: 운영자가 `/ops/sessions/:id/promote-to-directed` 명시
trigger + EditorialIntent lock 의무 (ADR-0025 INV-0025-4 와 일관).
auto-transition 없음. 같은 session 안에서 mode 변경 가능, 단 변경
이후의 round_synthesis 만 directed validation profile 적용.

대안 (fork session) 도 가능 — fork_kind='derivative' + 새 session mode
='directed'. 운영자 판단.
