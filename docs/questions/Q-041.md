# Q-041: updated_at 형식 CHECK constraint

## Context

PR #25 adversarial review iter 2 finding F-17. v0 운영 contract에서는
모든 writer가 strftime no-millis로 통일되어 있어 현재는 drift가 없지만,
새 writer가 `new Date().toISOString()` 형식을 사용하는 순간:

```
'2026-05-13T03:30:50.500Z' < '2026-05-13T03:30:50Z'  (lex 비교)
```

위 표현식이 TRUE가 된다 (`.` 0x2E < `Z` 0x5A). millis-bearing 행이 더
오래된 것으로 잘못 분류돼 stale-reclaim이 너무 적극적으로 동작한다.

## Discussion

해법은 storage-level CHECK constraint. SQLite는 `ALTER TABLE ... ADD
CONSTRAINT`을 지원하지 않으므로(SQLite는 ALTER로 ADD COLUMN / RENAME만
허용) 두 방식 중 하나를 채택:

**옵션 A — 새 schema 슬라이스에서 CHECK 포함한 채로 컬럼 생성 (greenfield용)**:

```sql
-- 컬럼 정의 안에 CHECK 인라인 (DEPLOY-1A.1 이전 시점, schema가
-- wipe-and-reseed 가능한 v0 단계에서 적용 가능)
CREATE TABLE discovery_queue (
  ...
  updated_at TEXT NOT NULL
    DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
    CHECK ( updated_at GLOB '????-??-??T??:??:??Z' ),
  ...
);
```

**옵션 B — 12-step table rebuild (production v0+ 이후)**:

```sql
-- https://www.sqlite.org/lang_altertable.html section 7
-- CHECK constraint를 ALTER로 추가할 수 없으므로 새 테이블에 CHECK를
-- 포함해 만들고, 데이터 복사 후 swap한다. PRAGMA foreign_keys = OFF
-- 와 함께 진행 후 다시 ON.
PRAGMA foreign_keys = OFF;
BEGIN;
CREATE TABLE discovery_queue_new (
  ...
  updated_at TEXT NOT NULL
    DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
    CHECK ( updated_at GLOB '????-??-??T??:??:??Z' ),
  ...
);
INSERT INTO discovery_queue_new SELECT ... FROM discovery_queue;
DROP TABLE discovery_queue;
ALTER TABLE discovery_queue_new RENAME TO discovery_queue;
-- 인덱스 재생성
CREATE UNIQUE INDEX ... ;
COMMIT;
PRAGMA foreign_keys = ON;
```

SQLite CHECK는 INSERT/UPDATE 시점에 검증되므로 wrong-format writer는
storage 가까이에서 차단된다.

### 현재 writer 포맷 stocktake (DEPLOY-1A.2 진입 전 정합 의무)

본 repo는 **두 가지 ISO 포맷이 코드베이스에 공존**한다. CHECK 도입 전에
포맷을 통일하거나 컬럼별로 다른 GLOB을 부여해야 한다 — 단일 GLOB을 모든
timestamp 컬럼에 부여하면 valid 쓰기가 fail한다 (PR #28 codex P2 지적).

| 컬럼 | 현재 writer | 포맷 |
|---|---|---|
| `discovery_queue.updated_at` | `snapshot-fingerprint.ts` SET_UPDATED_AT 상수 | strftime no-millis (`'%Y-%m-%dT%H:%M:%SZ'`) |
| `discovery_queue.discovered_at` | v6 schema DEFAULT | strftime no-millis |
| `crawl_state.last_polled_at` | `crawl-state.ts:35,116` `new Date().toISOString()` | **millis-bearing** (`'.sssZ'`) |
| `crawl_state.next_eligible_at` | `crawl-state.ts:125` `new Date().toISOString()` | **millis-bearing** |
| `raw_cache_items.expires_at` | `intervention-review.ts` `strftime('%Y-%m-%dT%H:%M:%fZ', ?, '+7 days')` | strftime with-millis (3+ digit fractional) |

### DEPLOY-1A.2 작업 가이드 (순서대로)

1. **포맷 통일 결정**: 모든 timestamp 컬럼을 millis-bearing으로 통일할지,
   no-millis로 통일할지, 컬럼별로 다른 형식을 유지할지 결정. **권고**:
   `Date#toISOString()`이 JS-side 표준이라 millis-bearing으로 통일하고
   `strftime('%Y-%m-%dT%H:%M:%fZ', ...)`로 SQL-side를 맞춘다. 단,
   `discovery_queue.updated_at`은 stale-reclaim의 lex 비교 정확도가
   millis 유무와 무관하므로 변환해도 동작에 영향 없음.

2. **writer 일괄 변경 PR** (DEPLOY-1A.2 진입 직전):
   - `snapshot-fingerprint.ts SET_UPDATED_AT` 상수를 `'%Y-%m-%dT%H:%M:%fZ'`
     로 변경 (or 반대 방향이라면 crawl-state.ts를 strftime no-millis로
     변경)
   - `crawl-state.ts:35,116,125`의 `new Date().toISOString()` 호출 검토
     (포맷 통일 결정에 따라 변경 or 유지)
   - `v6_discovery_queue.sql` DEFAULT 식 동일하게 변경

3. **CHECK constraint 적용**: 통일된 형식에 맞는 GLOB을 모든 컬럼에 적용.
   - no-millis 통일 시: `GLOB '????-??-??T??:??:??Z'`
   - with-millis 통일 시: `GLOB '????-??-??T??:??:??.???Z'` (3-digit 고정)
     또는 `GLOB '????-??-??T??:??:??.*Z'` (가변 fractional 허용)

4. **migration 적용 패턴**: 위 옵션 A (CREATE TABLE inline CHECK, v0
   greenfield) 또는 옵션 B (12-step table rebuild, post-deploy).

### 사용자 결정 필요 항목

1. CHECK constraint를 도입할지 (테스트가 새 형식 추가 시 더 시끄러워짐 — 의도된 변경에도 분류 작업 필요)
2. 도입한다면 어느 슬라이스에서 (DEPLOY-1A.2로 lock — 본 PR에서 이미 등록됨)
3. **포맷 통일 방향**: millis-bearing 일관 vs no-millis 일관 vs 컬럼별 분리
