# Q-041: updated_at 형식 CHECK constraint

## Context

PR #25 adversarial review iter 2 finding F-17. v0 운영 contract에서는
모든 writer가 strftime no-millis로 통일되어 있어 현재는 drift가 없지만,
새 writer가 `new Date().toISOString()` 형식을 사용하는 순간:

```
'2026-05-13T03:30:50.500Z' < '2026-05-13T03:30:50Z'  (lex 비교)
```

위 표현식이 TRUE가 된다 (`.` 0x2E < `Z` 0x5A). millis-bearing 행이 더
오래된 것으로 잘못 분류돼 stale-reclaim이 너무 적극적으로 동작한다.

## Discussion

해법은 storage-level CHECK constraint:

```sql
ALTER TABLE discovery_queue
  ADD CONSTRAINT updated_at_iso_no_millis
  CHECK ( updated_at GLOB '????-??-??T??:??:??Z' );
```

SQLite CHECK는 INSERT/UPDATE 시점에 검증되므로 wrong-format writer는
storage 가까이에서 차단된다.

### 적용 범위

- discovery_queue.updated_at (primary 대상)
- crawl_state.last_polled_at (이미 strftime 사용)
- raw_cache_items.expires_at (PR #25 iter 1 F-05에서 strftime millis로 통일됨 — 별도 CHECK)

### 사용자 결정 필요 항목

1. CHECK constraint를 도입할지 (테스트가 새 형식 추가 시 더 시끄러워짐 — 의도된 변경에도 분류 작업 필요)
2. 도입한다면 어느 슬라이스에서 (별도 schema-validator slice vs 다음 ALTER migration 묶음)

기본 권고: 다음 schema validator slice (Q-040 production-ready 직전)에서
모든 timestamp 컬럼에 일괄 적용.
