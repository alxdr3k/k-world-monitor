# Q-027: 백업 schedule + R2 lifecycle

## Context

ADR-0014 INV-0014-6에서 백업 schedule을 명시했지만 retention과 R2 lifecycle은
Q-027로 별도 finalize. R24 자체 audit에서 신설.

## Discussion

권고:

| 백업 종류 | 주기 | retention | 비고 |
|---|---|---|---|
| Neo4j dump (`neo4j-admin database dump`) | 일간 | 30d | 단일 인스턴스 SPOF — local + R2 양쪽 보관 |
| SQLite snapshot | 일간 | 90d | relational metadata + run_ledger 보호 |
| JSONL audit export | 월별 | 1y | human-readable trail (ADR-0014 INV-0014-7) |
| R2 derived artifact (자체 차트/export) | 무기한 | — | versioned by publish date |
| R2 open-license dataset | versioned | snapshot date per fetch | dataset_vintage 테이블과 일관 (ADR-0011 R9/Q4) |

운영 비용:
- Neo4j dump 일간 + 30d retention = R2에 30 객체 × 평균 dump size
- SQLite snapshot 일간 + 90d = 90 객체. SQLite는 단일 파일이라 작음
- JSONL export 월별 + 12d = 12 객체/년
