# Q-012: Neo4j ↔ SQLite sync 정책

## Context

ADR-0012로 graph object는 Neo4j, relational metadata는 SQLite로 분담했다. 두
store에 걸친 FK 관계가 있다:
- `policy_decisions.intervention_id` → `AccessIntervention.intervention_id`
  (SQLite → Neo4j)
- `ManualClaimEntry.intervention_id` → `AccessIntervention.intervention_id`
  (Neo4j → Neo4j, 같은 store)
- `metrics_run.run_id` → `run_ledger.run_id` (SQLite → SQLite)

cross-store FK는 sync 정책이 필요하다.

후보:
- (a) v0 batch sync (cron / GitHub Actions) — 일별 또는 시간별 SQLite ↔ Neo4j
  diff
- (b) Application-level dual-write — 모든 mutation을 두 store에 동시 적용
  (transactional 위험)
- (c) Neo4j CDC (5.13+) — change data capture로 실시간 sync (v2+ 검토)

## Discussion

- v0는 (a) batch sync 권고. 1인 운영 scale에서 sync lag 분 단위 허용.
- (b)는 transactional consistency 부담 — Neo4j와 SQLite는 서로 다른 transaction
  모델. 1인 운영에 과함.
- (c)는 Neo4j 5.13+ 의존. v0 영향 없음, v2+ 검토.

권고: v0 (a) batch sync. cron 주기는 1시간 또는 사용자 trigger.
