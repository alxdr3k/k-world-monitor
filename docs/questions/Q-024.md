# Q-024: Neo4j-specific 기능 활용 boundary

## Context

ADR-0014 Neo4j-native 기능 최대 활용 + 의도적 lock-in 수용 결정. R24 자체
audit에서 신설 — 구체적 boundary가 lock되어야 schema/query 설계가 일관.

후보 명세:

**APOC**:
- standard procedures: `apoc.load.json/csv`, `apoc.export.csv`, `apoc.periodic.iterate`,
  `apoc.text.*`, `apoc.date.*`, `apoc.path.expand` — 사용
- extended procedures (GraphQL endpoints, custom plugins) — v0 미사용

**GDS (Community Edition)**:
- Community 알고리즘 list — Louvain, Leiden, Label Propagation, Weakly
  Connected Components, PageRank, Betweenness, Closeness, Degree centrality,
  Node Similarity, K-NN, FastRP, Node2Vec — v1+ 활용
- Enterprise-only (PaR algorithms, advanced graph generators) — 미사용

**Cypher 5.x**:
- 사용: EXISTS / COLLECT / CALL {} subqueries / PERIODIC COMMIT / SHOW INDEXES
  / CONSTRAINTS / OPTIONAL MATCH / WITH chaining
- v2+ 검토: CDC (5.13+)
- 미사용: Cypher Enterprise extensions

## Discussion

- v0는 APOC standard + Cypher 5.x core로 시작 → schema 단순
- v1+에서 GDS 분석 진입 시 알고리즘 list 확정
- ADR-0014 INV-0014-2의 "vendor-neutral 제약 없음"과 일관

권고: 위 명세로 v0 lock. extended / Enterprise-only는 별도 ADR 권한 부여 시
도입.
