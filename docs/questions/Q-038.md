# Q-038: discovery_queue multi-worker worker_id CAS

## Context

PR #25 adversarial review (iter 1 finding F-15)에서 식별. v0는 단일
discovery worker가 cron-driven으로 한 번씩만 돌기 때문에 actual race가
거의 발생하지 않지만, ADR-0030의 INV-0030-2(serial write phase)는
"single-worker 가정"을 implicit하게 깔고 있다. multi-worker로 scale-out
하면 다음 sequence가 가능하다:

```
t=0:00  Worker A: BEGIN IMMEDIATE → claim row R as 'processing'
t=0:01  Worker A: heartbeat updates R.updated_at = T0
...
t=1:00  Worker A: 여전히 fetch 중 (Slow Neo4j)
t=1:00  stale-reclaim worker: R.updated_at < NOW-1h → 'pending'으로 reset
t=1:01  Worker B: BEGIN IMMEDIATE → claim row R as 'processing'
t=1:02  Worker A: heartbeat updates R.updated_at = T1 (succeeds — no worker_id check)
t=1:03  Worker A: stillProcessing check → true (Worker B claim 후이지만 status='processing'이라 통과)
t=1:04  Worker A: createSnapshotFingerprint(R) — duplicate work
t=1:05  Worker B: createSnapshotFingerprint(R) — duplicate work
```

## Discussion

### 후보 해법

| 옵션 | 비용 | 안정성 |
|---|---|---|
| (a) `discovery_queue.worker_id` 칼럼 + 모든 UPDATE WHERE에 `worker_id = $self` 조건 | 1 column + WHERE clause 변경 | high — 표준 CAS pattern |
| (b) heartbeat 주기 단축 + stale threshold 단축 | 0 schema change | medium — race window 좁아지지만 닫히지 않음 |
| (c) Redis distributed lock | 새 dependency | high — overkill for v0 |

(a)가 ADR-0030과 자연스럽게 호환된다. worker_id는 process-level UUID
(예: `worker_${ulid()}`)면 충분.

### 사용자 결정 필요 항목

1. v0를 명시적으로 "single-worker only"로 lock 할지 (Q dropped + ADR-0030에 invariant 추가)
2. INFRA-1B.3.x slice로 hardening promote 할지

기본 권고: (2). v0가 single-worker라도 stale-reclaim 자체가 "있는" 이상
race window는 0이 아니다. 다음 INFRA slice에서 worker_id 컬럼을 추가한다.
