# Q-037: AccessIntervention resolveIntervention importance_score race condition

## Context

PR #25 adversarial review (iter 1 finding F-08)에서 발견됨. v0 단일 operator
환경에서는 actual collision 확률이 낮아 deferred 처리했지만, ADR-0018이
"manual feedback inbound workflow"를 정의하면서 long-term에는 multi-reviewer
입력 channel(여러 운영자 / web UI / Slack bot 등) 가능성이 있어 명시적
locking 정책이 필요하다.

## Discussion

### 현재 코드

```typescript
// src/pipeline/feedback/intervention-review.ts:67-85
const result = await session.run(
  `MATCH (i:AccessIntervention {intervention_id: $interventionId})
   WHERE i.status = 'pending_user_review'
   SET ${setClause}
   RETURN count(i) AS matched`,
  { ... }
);
```

`setClause`가 importance_score adjustment(`+ $adjust`)를 포함할 때, 두 동시
트랜잭션이 MATCH의 WHERE을 통과하고 SET의 산술 연산을 각자 실행하면
`importance_score - 0.2`가 두 번 적용된다. Neo4j optimistic locking은 같은
property를 동시에 SET하는 경우 deadlock detection에 들어가지만, 산술 연산
(read-modify-write)은 별도 transaction visibility 문제다.

### 후보 해법

| 옵션 | 비용 | 안정성 | v0 영향 |
|---|---|---|---|
| (a) `apoc.lock.nodes([i])` | APOC standard 의존(이미 v0 stack lock — Q-024 resolved) | high | 없음 |
| (b) CAS pattern (RETURN prev, conditional SET) | Cypher 2-step | medium | 추가 round-trip |
| (c) SQLite advisory lock 외부화 | 새 storage layer | high | 별도 ADR 필요 |

(a)가 ADR-0018 + Q-024(APOC standard adoption) 컨텍스트에 가장 자연스럽지만,
multi-tenancy / cluster 도입 시 apoc.lock의 single-instance 제약과 충돌한다.

### 사용자 결정 필요 항목

1. v0 single-operator 가정을 명시적 invariant로 lock 할지 (= 본 Q를 dropped로 resolve)
2. 명시적 hardening slice를 v1+로 promote 할지 (= 본 Q를 in_progress로 두고 INFRA-1B.6.x로 분리)
3. apoc.lock vs CAS pattern vs 외부 lock 중 어느 옵션을 default로 채택할지

기본 권고: (2) + (a). 1-operator v0는 OK로 lock하고, INFRA-1B.6.x slice에서
apoc.lock.nodes를 도입한다.
