# Q-034: Auto retraction trigger 정책 v1+

## Context

DEC-005에서 v0 correction / retraction workflow를 manual approve로 lock
(ADR-0018 manual feedback CLI 기반). v1+ 에 어느 신호를 자동 trigger로 격상할
지 결정 필요. cite check 5+1 (ADR-0015) + access_intervention (ADR-0017) +
stale 트리거 (ADR-0010) 3종이 후보 신호.

## Discussion

신호별 위험 / 권고:

| 신호 | False positive 리스크 | 자동화 권고 | Pre-req |
|---|---|---|---|
| cite check claim 의존 retracted | 낮 (실제 retracted) | v1 auto → corrected | confirm UI |
| cite check claim 의존 stale | 中 (timing 기반) | v1 semi-auto → notification only | 운영자 review |
| access_intervention HIGH/CRITICAL on publication 의존 | 낮 | v1 auto → corrected | ADR-0017 cascade |
| access_intervention MEDIUM/LOW | 高 | v0/v1 manual 유지 | — |
| stale 트리거 (snapshot diff) | 中 | v1 semi-auto → notification | 운영자 review |
| stale 트리거 (counterclaim 등록) | 中 | v1 semi-auto → notification | 운영자 review |
| 외부 manual feedback (제보) | 中-高 | v0/v1 manual 유지 | — |

state 전이 자동화 권고:
- **live → corrected**: v1에 일부 신호로 자동 (cite check retracted, HIGH/
  CRITICAL intervention). 운영자 review 후 confirm. correction_ledger 자동
  append + 운영자 message 추가
- **corrected → retracted**: v2까지 manual 유지. retracted 는 사이트에서
  비공개 처리 → false positive 리스크 너무 큼. 사람 명시 승인만
- **모든 state 전이 후**: vault publications/ frontmatter `status` 수정 →
  git push → Cloudflare Pages build (DEC-006 trigger) → 사이트 visible
  반영
