  - **(1a) 분리 유지** — 현재 처럼 RSS/API crawler (b 의 일부) + ADR-0023

    interface 로 결과 emit.
  - **(1c) AI agent loop 도입** — 기존 single-shot tool-use (ADR-0023) 를
    multi-step agent loop 로 확장. LLM 이 internal-search.ts / external-

    반복 query → result → reasoning → next query. **별도 ADR 필요 — agent

  기본 권고: **(2a) + (2b) hybrid** — EditorialIntent.purpose 가 publication

  (Dossier 합성 mode). 두 layer 가 align 의무 — divergence 시 manual
  review.

  - **(3a) `research_round` 테이블 신규** — round_id (PK), session_id (FK
    research_session), parent_round_id (nullable, lineage), round_no,

  - **(3c) hybrid** — 시나리오 2 의 초기 round 는 research_round 테이블,

  기본 권고: **(3c) hybrid** — research_round 가 일반 lineage 테이블,
  ScenarioRevision 이 시나리오 evolution 전용. 둘 다 SUPERSEDES/UPDATES
  edge lineage 가능.

  ### (4) 라운드 간 context propagation

  - **(4a) System prompt cached prefix** — 라운드 1 의 conclusions_at_end
    + key promoted Claim summary 를 라운드 2 LLM 의 system prompt 의
    cached prefix 로 주입 (ADR-0023 prompt caching INV-0023-2 활용). 비용
    효율 + context window 절감.
  - **(4b) Dossier 직접 주입** — 라운드 1 의 Dossier 가 라운드 2 의

  - **(4c) Evidence pack reuse** — 라운드 1 의 build_evidence_pack 결과를
    라운드 2 시작 시 재로드. 새 query 만 다음 라운드에서.

  기본 권고: **(4a) + (4b) hybrid** — system prompt cached prefix 로 의미
  context (hypothesis evolution, key insights) + Dossier inheritance 로
  evidence anchor 누적. (4c) evidence pack 은 redundancy 위험 — 라운드 2

  ### (5) AI search 결과의 ingestion 강제

  - **(5a) 강제 ingestion** (GPT 권고) — AI search 가 발견한 URL 은 답변
