  부족한 부분 (사용자 지적):

  1. **개별 article URL (RSS item URL / manual intake URL) 의 conditional
     fetch 캐시**: 현재 crawl_state 는 RSS feed 단위로만 ETag / Last-Modified
     저장. RSS item 의 article URL 을 fetch 할 때는 conditional header 미사용
     — 재방문 시 매번 full body 다운로드.
  2. **재방문 주기 결정 알고리즘**: 현재 정책 없음. 한 번 Snapshot 만들면
     다시 fetch 안 함 (content_hash 변경 감지 안 됨).
  3. **변경 감지 알고리즘 — naïve hash diff 의 false-positive**: 광고
     trackback ID, timestamp, comment count, dynamic widget 등이 본문 외
     영역에서 매번 바뀌면 sha256 가 매번 다른 hash — semantic 변경 없는
     데도 "변경됨" 처리.
  4. **HTTP Cache-Control / Expires header 활용 부재**: 서버가 명시한
     캐시 hint 무시.

  사용자 결정 옵션:

  ### (1) 재방문 주기 (revisit policy)

  - **(1a) TTL-based**: 각 Source / Document 또는 Claim 에 `revisit_after`
    필드 추가 (default = source.reliability_tier 와 source_perspective 기반:
    `risk_observer` 는 짧게, `neutral` 은 길게). Snapshot 만든 후 N 일 후
    재 fetch.
  - **(1b) Change-rate adaptive**: Cho & Garcia-Molina (2003) "Effective Page
    Refresh Policies" 식 — 과거 K 번의 변경 감지 빈도 기반으로 다음 revisit
    주기를 estimate. 자주 바뀌는 source 는 짧게, 안 바뀌는 source 는 길게.
  - **(1c) Event-driven only**: 시간 기반 정책 없이, 다음 path 중 하나로만
    재 fetch:
    - manual `pipeline refresh <snap_id>` (운영자 명시)
    - cite check 가 stale 트리거 (시간 / counterclaim, ADR-0010)
    - publication preflight 직전 (모든 cited Snapshot 의 최신성 검증)
  - **(1d) Hybrid**: (1a) TTL 기본 + (1c) event-driven override. 가장 일반적.

  ### (2) 변경 감지 (change detection)

  - **(2a) Naïve sha256 diff**: 현재 구현 그대로. False-positive 많음.
  - **(2b) Content extraction + canonical text hash**: HTML 에서 본문만
    추출 (boilerplate / nav / ad 제거) 후 normalize 한 text 의 sha256.
    오픈소스: Mozilla Readability (browser engine), `@mozilla/readability`,
    Trafilatura (Python), `mercury-parser` (legacy mozilla).
  - **(2c) Locality-sensitive hashing (LSH) / SimHash**: 작은 변경 (오타
    수정, 단어 1~2 개 변경) 을 "같은 문서" 로 분류. Charikar (2002)
    SimHash. 오픈소스: `simhash-js`, `node-simhash`, Common Crawl 의 변경
    감지에서 사용.
  - **(2d) Shingling + Jaccard similarity**: w-shingle (w=3~5 word) + MinHash
    + Jaccard. 오픈소스: `minhash` (Python), `node-minhash`. 부분 표절 / 부분
    인용 감지에 강함.
  - **(2e) Hybrid (2b) → (2a)**: 먼저 content extraction 으로 본문만 추출
    후 sha256 diff. False-positive 대부분 제거. 추가 normalize: whitespace
    collapse, lowercase, punctuation strip.

  ### (3) HTTP cache header 활용

  - **(3a) 무시**: 현재 상태.
  - **(3b) ETag / Last-Modified 만**: article URL 마다 ETag / Last-Modified
    저장 후 다음 fetch 시 If-None-Match / If-Modified-Since header 전송. 304
    응답 시 body skip + content_hash 그대로 유지. 가장 표준적 (RFC 9110).
  - **(3c) Cache-Control max-age + Expires 도 활용**: server-driven cache
    hint 도 honor. (3b) 보다 적극적 — 서버가 "1일 유효" 라 했으면 1일 내
    재 fetch 안 함.

  ### 기본 권고

  - **(1d) Hybrid**: TTL 기본 (default 30일, source reliability_tier 와
    source_perspective 기반 조정 — risk_observer = 7일, neutral = 60일,
    opportunity_observer = 14일, mixed = 30일) + event-driven override
    (manual refresh / stale trigger / publication preflight).
  - **(2e) Hybrid**: Mozilla Readability (또는 Trafilatura — open license
    중 가장 정확도 높은 도구 선택) → normalize → sha256. False-positive
    감소 + 구현 단순성 균형.
  - **(3b) ETag / Last-Modified**: article URL 마다 저장. (3c) Cache-Control
    은 별도 슬라이스 (서버 hint 신뢰도 검증 필요).

  ### 추가 algorithm reference

핵심 문제:

- 외부 source 의 content 는 시간에 따라 변할 수 있음 (기사 수정, 통계 갱신,
  blog 글 retract). NFR-006 (snapshot durability via fingerprint) 도 변경
  감지 의무 명시.
- 그러나 모든 URL 을 매번 full body 재 fetch 는 비용 / 시간 / 서버 부담.
- 재방문 주기 + 변경 감지 알고리즘 + HTTP cache header 활용 3 layer 결정
  필요.

추가 고려:

- **NFR-003 (5-hop trace) 와의 관계**: 재방문 시 content 가 바뀌면 새 Snapshot
  생성 또는 기존 Snapshot 의 claim_status 가 `source_changed` 전이 (ADR-0011
  INV-0011-5). Publication 의 cite anchor 가 stale 또는 source_changed 면
  cite check 5+1 block — 재방문 정책이 publication freshness 직접 영향.
- **DEC-007 (retention / R2 lifecycle) 와의 관계**: raw_cache_items 24h~7d
  TTL 와 article-level cache 가 분리. raw_cache_items 는 임시 cache (extraction
  pipeline 안에서만), article-level snapshot 은 영구. 두 layer 의 분리는
  명확.
- **Q-001 (scenario horizon enum) + Q-002 (Dossier stale_after default)
  와의 관계**: 재방문 주기 default 가 stale_after default 와 align 의무.
  Q-002 resolution 시점에 본 Q 의 default 도 같이 lock.
- **robots.txt revisit**: safe-fetch 의 robots cache 24h TTL — 재방문 시점
  마다 robots.txt 재 fetch (이미 구현). 본 Q 의 revisit 와 별도.
- **GitHub Actions cron 일일 1회 단일 실행 가정 (DEC-013)**: revisit 후보
  URL 들이 daily candidate cap 20 안에 들어와야 함 — TTL 30일 default 에서
  소스 72개면 daily 평균 2.4건 revisit 후보 (소스 distribution + reliability
  tier 보정 후) — daily cap 안에 충분.

## 사용자 결정 필요 항목

1. **재방문 주기 정책 (1a/1b/1c/1d)** — 기본 권고 (1d) hybrid 동의 여부.
   TTL default 값 (30일) 과 source_perspective 별 조정 (risk 7일 / neutral
   60일 / opportunity 14일 / mixed 30일) 동의 여부.
2. **변경 감지 알고리즘 (2a/2b/2c/2d/2e)** — 기본 권고 (2e) Mozilla
   Readability + normalize + sha256 동의 여부. False-positive 감소 정도
   를 측정할 수 있는 SPIKE 도입 시점.
3. **HTTP cache header 활용 (3a/3b/3c)** — 기본 권고 (3b) ETag / Last-
   Modified 만. (3c) Cache-Control 까지 갈지 여부.
4. **content extraction 도구 선택** — Mozilla Readability (Apache 2.0, JS
   네이티브, 본 repo bun 환경 호환) vs Trafilatura (GPL-3.0, Python — 본
   repo Polars 등 Python stack 안에서 호환 가능하지만 vendor surface 추가).
   기본 권고: Readability (JS native, license 호환).
5. **구현 슬라이스 우선순위** — INFRA-1B.8 (revisit policy) / INFRA-1B.9
   (content canonical) / INFRA-1B.10 (conditional fetch) 의 어느 것을 먼저
   진입할지. 기본 권고: INFRA-1B.10 (conditional fetch) 먼저 — 가장 단순
   + 가장 즉시 비용 절감. 그다음 INFRA-1B.9 (canonical text hash). 마지막
   INFRA-1B.8 (revisit policy) — TTL / event-driven 결합 layer.
6. **2주 P0-M6 turn-key MVP lock 안에서 본 Q 의 구현 슬라이스 진입 시점**
   — P0-M3 (Extraction & Review) 이전 / 안 / 이후 결정. 기본 권고:
   INFRA-1B.10 은 P0-M2 게이트 검증 안에 흡수 (소량 변경), 나머지 2개는
   P1+ (turn-key MVP 후).
