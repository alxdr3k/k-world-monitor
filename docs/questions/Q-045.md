# Q-045: invariant validator scope 확장 — ADR INV 본문 ↔ 코드 enforcement cross-ref 정적 검증

## Context

본 Q 는 2026-05-13 comprehensive review 의 사용자 결정 필요 항목 #4 이다.

ADR-0002 invariant tracking system (`docs/adr/0002-invariant-tracking-system.md`)
은 doc-doc 정합성 (scope, term, invariant policy) 을 검증한다. 그러나
ADR INV 본문이 "코드에서 다음과 같이 강제된다" 라고 주장하면 그 주장
자체는 코드 grep / static check 없이는 verifiable 하지 않다.

본 review 에서 발견한 NFR-008 audit log 누락 (Q-044) 은 INV-0012-3 본문
주장 ("R2 upload 시 policy_decisions ledger row 작성") 이 코드에서 누락
되어 있는데도 validator 가 잡지 못한 사례다. 향후 ADR INV 가 추가될
때마다 같은 silent drift 가 발생 가능.

## Discussion

위 frontmatter `proposed_answer` 의 옵션 (a) ~ (c) 와 기본 권고 (b) 가
적혀 있다. 추가 고려:

- (b) 채택 시 ADR frontmatter schema 가 변경됨 — boilerplate 와의 sync
  관계 검토 필요. `AGENTS.policy.md` 가 boilerplate 소유 정책이므로
  cross_ref_code 추가는 boilerplate PR 로 먼저 lock 후 본 repo 가
  채택하는 흐름일 수도. 또는 본 repo local extension 으로 lock 후
  boilerplate 로 promote 검토.
- cross_ref_code 의 형식 — file:line 은 fragile (코드 이동 시 stale).
  export name 은 stable 하지만 anonymous function / inline check 는
  cover 못함. tradeoff 결정 필요.
- 검증 timing — validator 는 PR 마다 advisory 실행. cross_ref_code stale
  도 warning 이지만 보고가 누적되면 validator 자체가 noise 됨.

기본 권고 (b) 의 작업 순서:

1. `docs/templates/INVARIANT_ENTRY.md` (또는 등가 위치) 에 cross_ref_code
   필드 schema 추가.
2. `scripts/validate_invariants.ts` 에 cross_ref_code 검증 로직 추가 —
   각 entry 가 file:line 형식이면 line 존재 검사, export name 형식이면
   grep 검사.
3. INV-0012-3, INV-0028-1~6, INV-0023-3 부터 cross_ref_code 채움 (점진적
   backfill).
4. validator 출력에 cross_ref 검증 결과 섹션 추가.

## 사용자 결정 필요 항목

1. validator scope 확장을 채택할 것인가 — (a) / (b) / (c) 중 선택
2. cross_ref_code 형식 — file:line (precise) / export name (stable) /
   둘 다 허용 중 선택
3. boilerplate 와의 sync 정책 — local extension 으로 먼저 lock vs
   boilerplate PR 먼저 lock
4. 채택 시 우선 backfill 대상 INV 우선순위 — 본 review 발견 기준 (INV-0012-3
   raw_cloud / INV-0028-* safe-fetch / INV-0023-3 domain_override_reason)
   동의 여부
