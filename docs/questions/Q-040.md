# Q-040: v0→production schema migration framework

## Context

PR #26 review에서 Codex P1 코멘트 3건:
1. `migrate.ts` v7 drop — preexisting v6 DB의 updated_at 칼럼 누락
2. `snapshot-fingerprint.ts` — 기존 queue rows의 slug-form source_id 미정규화
3. `run-discovery.ts` — `crawl_state.source_id` slug→canonical remap 없음

위 3개는 모두 "production v6 DB가 이미 deployed 됐다"는 가정을 깐다.
사용자가 명시한 v0 운영 contract("schema 부담 없음 / 마이그레이션 불필요")
가 그 가정을 부정하므로 v0에서는 정당. 그러나 production deploy 직전
시점에는 정식 framework 필요.

## Discussion

### v0 운영 contract (이번 PR에서 명문화)

- `research.db`는 매 schema 변경마다 wipe 가능
- 모든 base schema 파일은 deploy 전까지 re-writable
- `bun run seed:sources`로 source_registry 재구성
- 기존 queue rows는 drop accepted

### Production-ready 시점 (deploy slice 진입 직전)

- 마지막 wipe-and-reseed 시점 stamp + 그 이후 모든 변경은 ALTER로만
- Backfill 스크립트 의무화 (slug-form id remap, updated_at 칼럼 추가 등)
- Migration runner에 rollback 지원 (BEGIN/ROLLBACK envelope은 이미 있음)
- DB schema diff CI check (drift detection)

### 사용자 결정 필요 항목

1. v0 contract를 어디에 명문화할지 — `docs/05_RUNBOOK.md`, `docs/current/OPERATIONS.md`,
   또는 새 ADR
2. production-ready 진입 시점을 어느 milestone로 lock 할지 (M6 publication?
   별도 DEPLOY milestone?)
3. backfill scripts의 location/규약 — `migrations/sqlite/backfill/`?

기본 권고: (a) v0 contract를 `docs/05_RUNBOOK.md`에 명문화, (b)
production-ready entry를 별도 DEPLOY-1A milestone으로 분리, (c) backfill은
M6 publication milestone 직전에 별도 slice.
