# Q-048: CI branch protection — invariant-check / ci workflow 를 advisory 에서 required check 로 promote 할지

## Context

본 Q 는 2026-05-13 comprehensive review 의 사용자 결정 필요 항목 #7 이다.

본 comprehensive review 가 발견한 doc-code drift 8 건 중 다수가 "When
changing code" 규칙 위반 누적 — code 변경 시 thin docs 동반 갱신 0회.
원인의 일부는 CI 가 사실상 비활성 상태였던 것. 본 PR 이 CI workflow 를
rename 으로 advisory 활성화하면서 표면적으로 PR check 가 동작하지만,
branch protection required 등록 없이는 self-bypass 가능.

## Discussion

위 frontmatter `proposed_answer` 의 옵션 (a) ~ (d) + 기본 권고 (b) 가
적혀 있다. 추가 고려:

- ADR-0002 INV-0002-1 "invariant validator 는 warning-level by contract"
  — invariant-check 를 required 로 promote 하면 contract 위반. (c) 는 reject.
- 1 인 운영자 + main 직접 push 허용 정책 (`docs/01_PRD.md:208` "main 브랜치
  직접 push 허용") 과 branch protection required check 의 관계 — 직접
  push 는 required check 우회 가능. PR-based workflow 사용 시점에만 의미
  있음. 본 Q 는 PR-based 인 경우만 enforce.
- typecheck 는 fast (~수 초). bun test 는 18 file 200+ 케이스 — Bun
  native 로 1 분 이내 expected. CI 비용 측면 acceptable.

기본 권고 (b) 의 작업 순서 (사용자 결정 후):

1. GitHub UI 또는 `.github/branch-protection.yml` (해당 보일러플레이트
   파일이 있으면) 에서 main branch protection 설정.
2. Required check 로 `ci.yml` 의 jobs (build + test + typecheck) 등록.
3. `invariant-check.yml` 은 advisory 유지 (status check 표시되지만
   merge 차단 안 함).
4. `docs/current/TESTING.md` 의 "CI / required checks" 표 갱신.

## 사용자 결정 필요 항목

1. CI required check 등록 정책 — (a) 모두 advisory / (b) `ci.yml` 만
   required / (c) 모두 required / (d) `ci.yml` 의 일부 job 만 required
   중 선택
2. main 직접 push 정책 (`01_PRD.md:208`) 과 PR-only 정책 충돌 검토 —
   본 Q resolution 이 PR-only 로 전환을 의미하는지, 아니면 직접 push 는
   여전히 허용하면서 PR 만 enforce 하는지 결정
3. required check 등록 시 본 PR 안에서 작업할지, 별도 PR 로 분리할지
   (기본 권고: 별도 PR — branch protection 변경은 별도 commit 으로 기록
   하는 게 audit-friendly)
