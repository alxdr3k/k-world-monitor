# Q-046: Tier 0 호출 일일 quota 5회 (DEC-010 cap) 의 코드 enforcement 위치

## Context

본 Q 는 2026-05-13 comprehensive review 의 사용자 결정 필요 항목 #5 이다.

DEC-010 의 cost ceiling 은 4 layer 로 구성:
- soft $5 / 일
- hard $7.5 / 일
- weekly $25
- Tier 0 일일 cap 5 회
- backfill 별도 budget bucket

OPS-1A.1 (run ledger) 가 cost 합산은 구현했지만 Tier 0 cap 은 미구현
상태. P0-M3 EXTR-1A.* 슬라이스 진입 시 cross-vendor review 호출이
시작되면서 Tier 0 호출이 발생 — 본 Q 가 resolved 되지 않으면 cap 위반
silent drift 위험.

## Discussion

위 frontmatter `proposed_answer` 옵션 (a) ~ (c) + 기본 권고 (a) +
OPS-1A.2 흡수가 적혀 있다. 추가 고려:

- cap 위반 동작 — (i) throw / (ii) warn / (iii) backoff. backoff 는
  cron 환경에서는 다음 실행 시 retry 되므로 자연 throttling 효과. throw
  는 명시적이지만 운영자가 결정해야 함. v0 turn-key MVP 시점은 daily
  manual operation 이므로 throw 가 사람-친화적.
- weekly $25 / soft $5 / hard $7.5 의 enforcement 도 별도 결정 필요. 본
  Q 는 Tier 0 cap 만 다루지만 cost ceiling 의 다른 3 layer 도 같은
  enforcement 패턴 필요 — 본 Q resolution 시 함께 다룰지, 별도 Q-### 로
  분리할지 결정.

기본 권고 (a) 의 작업 순서:

1. `src/ops/run-ledger.ts` 에 `getDailyTier0Count(date)` 함수 추가
   (status='completed' AND tier=0 조건 COUNT).
2. `assertTier0Available()` 함수 추가 — 본 함수가 cap 5 초과 시 throw.
3. `startRun()` 안에서 `if (input.tier === 0) assertTier0Available();`
   추가.
4. `tests/unit/run_ledger_tier0_cap_test.ts` 신규 — TEST-019 확장 또는
   별도.
5. AC-019 본문에 Tier 0 cap 측정 추가.

## 사용자 결정 필요 항목

1. Tier 0 cap enforcement 위치 — (a) startRun 내부 inline / (b) OPS-1A.2
   throttling worker / (c) metrics alert 만 중 선택
2. cap 위반 시 동작 — (i) throw / (ii) warn + continue / (iii) backoff
3. 본 Q 가 Tier 0 cap 만 다룰지, weekly $25 / soft $5 / hard $7.5 의
   다른 ceiling 도 같이 다룰지 결정 — 기본 권고: Tier 0 만 본 Q,
   나머지는 별도 Q (또는 OPS-1A.2 슬라이스 안에 묶음)
