# DEC-015: SQLite 마이그레이션 규율 — 런타임 CREATE TABLE IF NOT EXISTS 금지

## Context

Opus 아키텍처 리뷰 Q2 발견: `src/storage/source-registry/seed.ts:154-159` 가
`source_registry_slug_map` 테이블을 런타임 `CREATE TABLE IF NOT EXISTS` 로 생성한다.
이는 `migrations/sqlite/v1_schema.sql` 이 확립한 "마이그레이션 파일이 스키마 소스"
원칙에 위반하며, `schema_migrations` 버전 레저에서 추적되지 않아 스키마 드리프트가 발생한다.

## Decision

**모든 SQLite DDL 은 `migrations/sqlite/*.sql` 파일로만 관리한다.**
애플리케이션 코드(`src/`)에서 `CREATE TABLE`, `CREATE INDEX`, `ALTER TABLE` 직접 호출은 금지한다.

### 즉시 적용 (INFRA-1B.1.x hotfix):

1. `migrations/sqlite/v3_source_registry_slug_map.sql` 파일 생성:
   ```sql
   -- v3_source_registry_slug_map
   CREATE TABLE IF NOT EXISTS source_registry_slug_map (
     slug      TEXT PRIMARY KEY,
     source_id TEXT NOT NULL
   );
   ```
2. `scripts/migrate.ts` 의 버전 체크에 v3 추가.
3. `seed.ts` 의 인라인 `CREATE TABLE IF NOT EXISTS source_registry_slug_map` DDL 제거.
   마이그레이션이 완료된 DB 를 전제하고 테이블 존재를 가정한다.
   (PRE-0028-2 와 동일한 패턴 — `source_material_policy` 는 이미 이 방식)

## Rationale

- 마이그레이션 파일이 스키마 단일 소스이면 DB 재구축, 테스트 픽스처, 프로덕션 배포가
  동일한 경로를 사용하게 된다
- `schema_migrations` 버전 레저에서 모든 스키마 변경이 추적됨
- `seed.ts` 의 인라인 DDL 제거로 "시드 전 마이그레이션 필요" 계약이 명확해짐

## Consequences

- 긍정: 스키마 드리프트 차단, 마이그레이션 이력 완전성
- 부정: `seed.ts` 를 그냥 실행하면 마이그레이션 미실행 시 실패 — 이미 PRE-0028-2 의 preflight guard 가 `source_material_policy` 에 대해 동일하게 동작
- Follow-ups: Slice `INFRA-1B.1.x` 에서 적용
