# DEC-021: Q-049 일괄 resolution

## Context

Q-049 (재방문 정책 + 변경 감지 알고리즘) 의 5 개 사용자 결정 항목 중 #1, #2
는 2026-05-13 결정 lock (GPT 답변 + 비판적 메타 리뷰 + 본 메타 보강 모두

남은 #3, #4 는 2026-05-14 사용자 GPT 리뷰 후 답변 — 본 DEC 가 5 개 항목
일괄 lock + Q-049 status: open → resolved 전환.

## Decision

### [Q-049 #1] 재방문 주기 정책

채택: **Hybrid TTL + event-driven** (content type 기반 default).

- 새로 발견된 article / blog / 보도자료: 24~72h 내 1회 재검증
- 1회 안정 확인된 일반 article: 7d
- 2~3회 연속 unchanged: 30d
- 연구보고서 / PDF / 기관 리포트: 30~90d
- 공식 통계 / API: URL 재방문 X — dataset_vintage / release calendar 중심
- 실제 publication 인용 Snapshot: TTL 무관 — publication preflight 직전
  conditional validation 강제

event-driven trigger:
- `manual_refresh` (운영자 명시)
- `stale_trigger` (시간 / counterclaim, ADR-0010)
- `publication_preflight`
- `multi_round_force_revalidate` (Q-050 cross-cutting)

source_perspective 기반 권고 reject — perspective 는 발행 편향 분류이지
변경 주기 proxy 가 아님 (GPT 정정 + 본 메타 리뷰).

### [Q-049 #2] 변경 감지 알고리즘

채택: **Mozilla Readability + linkedom + normalize + sha256 = `canonical_text_hash`** + **`raw_body_hash` 별도 보존**.

- raw_body_hash = 원본 bytes sha256 (현재 content_hash 와 동일)
- canonical_text_hash = HTML 본문 추출 (Readability) → normalize → sha256
- PDF / API / CSV: canonical_text_hash 미사용 — raw hash 또는 dataset
  vintage checksum
- SimHash / MinHash 는 P1+ phasing (MVP canonical exact hash 만)
- DOM parser = **linkedom** (lighter than jsdom)
- content extraction tool = **Mozilla Readability (Apache-2.0)** — Trafilatura
  (Apache-2.0, Python) 는 P1+ Data Science Module 도입 후 검토

### [Q-049 #3] Snapshot Neo4j schema UNIQUE constraint

채택: **옵션 A — `raw_body_hash` 만 UNIQUE, `canonical_text_hash` 는
nullable + non-unique INDEX**.

```cypher
// 기존 (rename only — content_hash 의 의미를 명시)
CREATE CONSTRAINT snap_raw_body_hash_unique
  FOR (s:Snapshot) REQUIRE s.raw_body_hash IS UNIQUE;

// 신규 (P1+ INFRA-1B.9 에서 ALTER 추가)
CREATE INDEX snap_canonical_text_hash_idx
  FOR (s:Snapshot) ON (s.canonical_text_hash);
```

**근거**:
1. NFR-006 의 fingerprint 정의 (byte-level) 보존. ADR-0012 INV-0012-1
   (snapshot fingerprint = URL+content_hash+locator) 의 의미를 그대로
   유지.
2. wrap diff (광고 / timestamp / tracker) 의 false-positive 회피는
   schema-level 이 아닌 **trigger 정책 layer (#4 채택)** 에서 분리.
3. Implementation 단순 — 기존 `MERGE (s:Snapshot {content_hash: ...})`
   path 가 raw_body_hash rename 만으로 동작.

옵션 B (둘 다 UNIQUE) + 옵션 C (composite UNIQUE) 는 reject —
byte-level identity 흐려짐 / mirror 사이트 dedup 충돌 위험.

**Migration sequence (GPT 구현 주의 반영, INFRA-1B.9 슬라이스)**:

```
1. 기존 Snapshot 의 content_hash → raw_body_hash 컬럼 backfill
   (SET s.raw_body_hash = s.content_hash WHERE s.raw_body_hash IS NULL)
2. snap_content_hash_unique constraint DROP
3. snap_raw_body_hash_unique constraint CREATE
4. canonical_text_hash property 추가 (nullable, INDEX non-unique)
5. 신규 Snapshot 부터 Readability 적용해 canonical_text_hash 계산
6. 기존 Snapshot 의 canonical_text_hash backfill 은 별도 batch job
   (host concurrency cap 유지, daily candidate cap 20 와 별도)
```

### [Q-049 #4] ADR-0010 snapshot_diff 정의 reflow 위치

채택: **옵션 (i) — ADR-0010 본문 직접 갱신 (INV-0010-4 추가)** + 본
DEC-021 으로 cross-ref.

ADR-0010 INV-0010-4 신규 추가 (`docs/adr/0010-stale-triggers-and-review-
throttling.md`):

```yaml
- id: INV-0010-4
  statement: INV-0010-1 (b) snapshot diff 의 정의 = canonical_text_hash
    diff (의미 변경 trigger). raw_body_hash diff 만 단독으로는 stale
    trigger 가 아니다 — wrap diff (광고 / timestamp / tracker) 의 false-
    positive 회피. raw diff + canonical 동일 인 경우 source_changed_minor
    alert 로 분류 (v1+). canonical_text_hash NULL Snapshot (P0 시점,
    INFRA-1B.9 도입 전) 은 fallback 으로 raw_body_hash diff 적용. Q-049 /
    DEC-021 (2026-05-14) 결정
  status: active
```

ADR-0010 Decision 표의 snapshot_diff row 도 갱신 — canonical 우선 + raw
fallback 명시.

**근거**:
1. ADR-0010 = stale trigger 정의 single source of truth — 구현자 / reviewer
   가 ADR-0010 만 보면 trigger 정책 완전 이해 가능.
2. 의미 변경 (raw → canonical 분리) 이 ADR scope 안 — 별도 ADR supersede
   까지 가지 않음. INV 추가가 audit-friendly.
3. DEC-021 (본 결정) 이 historical anchor, ADR-0010 본문이 canonical text
   — 양립 가능.

옵션 (ii) 별도 ADR + (iii) DEC-only 는 reject — ADR-0010 본문에 정의
누락 시 구현자 cross-ref 의무 + ADR-0010 단독 read 시 의미 누락 risk.

### Migration order 의 의무

GPT 의 구현 주의 명시 채택 — backfill 순서:

1. content_hash → raw_body_hash backfill 먼저 (모든 기존 Snapshot)
2. 그다음 raw_body_hash UNIQUE constraint 생성 (DROP old + CREATE new)
3. canonical_text_hash nullable property 추가 + non-unique INDEX
4. 신규 Snapshot 부터 Readability 적용
5. 기존 Snapshot canonical_text_hash backfill 은 별도 batch (낮은
   우선순위, 별도 throttle)

본 순서 위반 시 (예: UNIQUE constraint 먼저 생성) 기존 row 의 raw_body
_hash NULL → constraint violation. INFRA-1B.9 slice 구현 시점 sequence
의무.
