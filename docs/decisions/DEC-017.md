# DEC-017: Safe-Fetch 컨텐츠 종류별 최대 바이트 상한

## Context

ADR-0028 INV-0028-4 가 컨텐츠 종류별 바이트 상한을 요구하지만 구체 값은 DEC 로 위임한다.
zip bomb / 무제한 body 방어를 위해 safe-fetch 가 압축 해제 바이트를 카운트하며
상한 초과 시 연결을 끊는다.

## Decision

| 컨텐츠 종류 | 최대 바이트 (압축 해제 기준) | 비고 |
|---|---|---|
| HTML 페이지 | 10 MB | 일반 기사 페이지 |
| RSS / Atom XML | 5 MB | 피드 파일 |
| JSON API 응답 | 5 MB | REST/GraphQL API |
| PDF 문서 | 20 MB | 보고서 / 연구 자료 |
| 데이터셋 (CSV / JSONL) | 50 MB | ADR-0024 Data Science Module 입력 |
| 기타 / 불명 | 1 MB | 보수적 기본값 |

zip bomb 비율 상한: 압축 해제 / 압축 = 100:1 초과 시 즉시 중단 (컨텐츠 종류 무관).

`safe-fetch` API:
```typescript
export const MAX_BYTES = {
  html: 10 * 1024 * 1024,
  rss:   5 * 1024 * 1024,
  json:  5 * 1024 * 1024,
  pdf:  20 * 1024 * 1024,
  dataset: 50 * 1024 * 1024,
  unknown: 1 * 1024 * 1024,
} as const;
```

## Rationale

- 일반 기사/피드 크기 분포 기준으로 정상 컨텐츠를 수용하면서 비정상 payload 차단
- 데이터셋 50MB 는 ADR-0024 의 Polars/DuckDB 입력 범위 내 — R2 업로드는 별도 경로
- 불명 컨텐츠 1MB 보수적 기본값으로 미분류 공격 대응

## Consequences

- 긍정: zip bomb / OOM 공격 차단, 메모리 사용량 예측 가능
- 부정: 간혹 큰 합법적 피드 (10MB+ 아카이브 RSS) 가 잘릴 수 있음 — 운영 로그로 모니터링
- Follow-ups: Slice `INFRA-1B.2a` 에서 safe-fetch 구현 시 이 상한 적용
