# DEC-014: SQLite busy_timeout=5000ms PRAGMA 표준

## Context

Opus 아키텍처 리뷰 Q2 발견: `src/storage/sqlite/connection.ts` 의 `getDb()` 에
`busy_timeout` PRAGMA 가 없다. WAL 모드에서 단일 프로세스 내부는 JS 단일 스레드
특성상 write 경쟁이 없지만, 향후 discovery worker 가 별도 프로세스로 분리되거나
`bun run migrate` 가 worker 실행 중 호출되면 `SQLITE_BUSY` 가 즉시 발생한다.
현재 retry wrapper 도 없다.

## Decision

`src/storage/sqlite/connection.ts` 의 `getDb()` 에 아래 PRAGMA 를 추가한다:

```typescript
db.run("PRAGMA busy_timeout = 5000");
```

기존 WAL / foreign_keys / auto_vacuum PRAGMA 와 함께 DB open 직후 실행.

## Rationale

- `busy_timeout = 5000` ms 는 단발성 lock 충돌을 retry 없이 자동 대기 처리 — 코드 변경 최소화
- WAL 모드에서 writer 1개, reader 다수이므로 5초 대기는 실용적 상한 (죽은 writer 는 checkpoint 로 해소)
- SQLite 자체 기능이므로 외부 의존성 없음

## Consequences

- 긍정: 다중 프로세스 시나리오에서 즉시 `SQLITE_BUSY` 대신 5초 대기 후 재시도
- 부정: 없음 (단일 프로세스 현재 상황에서도 무해)
- Follow-ups: Slice `INFRA-1B.1.x` 에서 함께 적용
