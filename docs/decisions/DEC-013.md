# DEC-013: Active source ingestion cap + v0 운영 원칙

## Context

GPT 메타 리뷰 (2026-05-11) 의 3 순위 병목 + 전략적 위험 1 지적:

> 3순위 병목: source ingestion 의 과범위 ... 특히 공식 RSS/API source 는
> item 이 많고, 대부분은 콘텐츠화할 가치가 없습니다. 따라서 Collection
> Queue 단계에 "수집" 보다 더 중요한 것은 triage 입니다. ... 이 cap 이
> 없으면 "잘 수집되지만 아무것도 발행하지 못하는" 시스템이 됩니다.

> 위험 1: "시스템을 만들다가 글을 못 쓰는" 상태 ... P0-M6 의 첫 publication
> 이 너무 늦어지면 프로젝트는 실패합니다. 완벽한 graph 보다 불완전하지만
> traceable 한 첫 글이 더 중요합니다. 추천은 명확합니다: 첫 publication
> 전에는 새 ADR/DEC/Q 를 최소화하고, 경제 카테고리 active source subset
> 으로 실제 글 1개를 만드는 데 집중해야 합니다.

본 repo 의 비판적 리뷰 = **수용**. 이유:
- 운영 cap 없으면 LLM cost ceiling (DEC-010 soft $5 / hard $7.5) 보다 운영
  자 attention cost 가 먼저 터짐 — 1인 운영 환경 차단 필요
- "수집 vs triage" 비대칭 — 공식 RSS/API source 의 item 대부분이 promoted
  claim 가치 없음. Discovery queue 진입을 좁히는 게 핵심
- 첫 publication 전 결정 도입 최소화는 "문서/구조가 제품 검증 앞서가는"
  본 repo 의 risk 직접 대응

## Decision

### 1. Active source ingestion cap (운영 정책)

| 단계 | cap | 기준 | enforcement |
|---|---|---|---|
| **일일 candidate item** | 20개 | Discovery polling 결과 raw item 중 collection queue 진입 후보 | `INFRA-1B.2` worker — 일일 score-rank top 20 만 queue 진입. 초과 시 drop + 다음 일 priority lift |
| **일일 promoted claim** | 5~10개 | extract 후 reviewer queue 통과 + promoted | `EXTR-1A.4` (review queue) — 일일 promoted 10 초과 시 reviewer 알람. 5 미만이면 candidate item 풀이 작거나 source quality 낮음 신호 |
| **weekly dossier** | 1개 | 1 주제 / 1 글 단위 dossier 합성 | `AGG-1A.1` (Dossier composer) — 운영자 명시 trigger. 자동 다수 합성 X |
| **weekly publication** | 1개 (v0 turn-key) | publication 발행 빈도 | `PUB-1A.5` 의 v0 cadence. v1+ 에 2~3 / week 검토 |

### 2. v0 운영 원칙 (3 종)

#### 원칙 1: 첫 publication 전 새 ADR/DEC/Q 최소화

- 첫 publication (PUB-1A.5 의 한국 부동산 폭락 시나리오, DEC-011) 발행
  전에는 **새 ADR / DEC / Q 도입 최소화**.
- 새 결정이 필요해 보이면 우선 issue / TODO 메모 (vault 또는 issue tracker)
  로 보관하고 첫 발행 retrospective 후 정식 ADR/DEC/Q 로 승격.
- 예외: 첫 발행 자체를 차단하는 결정 (예 INFRA-1A.2 Cypher schema, EditorialIntent
  enum, evidence_role enum 등) 은 즉시 lock.

#### 원칙 2: 수집보다 triage

- Discovery polling 의 raw item 풀에서 **5~10% 만** promoted claim 후보로
  진입 (의도된 narrow funnel).
- Collection queue 단계의 LLM-judge (Tier 3 GPT-5 nano) 또는 운영자 review
  로 triage. score-rank 알고리즘 (예 source reliability_tier × extraction
  confidence × topic relevance) 으로 priority 정렬.
- 일일 candidate cap 20개 안에서 triage — 초과 시 next-day priority lift.

#### 원칙 3: 첫 글은 semi-manual 권장

- LLM extractor / Dossier composer / Scenario validator / Thesis composer
  가 완전 자동 동작 안 해도 OK.
- 운영자가 active source 5~8개 직접 read → claim 수동 등록 → Dossier/
  Scenario/Thesis semi-manual 작성 → publication.
- 첫 목표 = "자동화율 max" 아니라 "이 구조로 좋은 글이 나오는가 검증".
- 첫 발행 retrospective 에서 자동화율 trade-off 평가 후 v0.5 / v1+ 단계
  점진 자동화.

### 3. Cap enforcement metric (REQ-024 / ADR-0020 확장)

신규 metric (운영 cap monitoring):
- `daily_candidate_item_count` (Discovery worker)
- `daily_promoted_claim_count` (Review queue)
- `weekly_dossier_count` (Dossier composer)
- `weekly_publication_count` (Publication ledger)

cap 초과 시 metrics_alerts 테이블에 row insert (운영 알람).

### 4. 첫 publication 전 새 결정 최소화 — 본 DEC 자체가 마지막 의무 결정

DEC-011 / ADR-0026 / ADR-0027 / DEC-012 / DEC-013 (본 DEC) 까지가 첫 발행
전 의무 결정. 이후 새 ADR/DEC/Q 는 첫 발행 retrospective 후 도입.

단 다음 결정은 첫 발행 차단 — 즉시 lock 의무 (예외):
- INFRA-1A.2 Cypher schema migration (Source `active_v0` + Claim
  `:EVIDENCE_FOR` + Thesis `:HAS_INTENT` + ContentDraft `editorial_
  intent_id` 등 schema lock)
- Q-021 active subset 73개 → 75개 (한국 부동산 specific KAB/MOLIT/HF 추가
  등록) accept
- EditorialIntent sample → 실제 PUB-1A.5 진입 시 운영자 명시 작성 / lock
- AC-036~042 운영자 verify checklist (DEC-012) 첫 적용

## Rationale

- 운영 cap 명시 = "수집되지만 발행 안 되는" 시스템 차단
- 첫 발행 전 결정 최소화 = "시스템 만들다가 글 못 쓰는" 상태 방지 (GPT
  메타 리뷰 위험 1)
- semi-manual 권고 = LLM extractor 미완성도 첫 발행 가능 — 운영자가 검증
  loop 의 master loop 역할
- 운영 metric (REQ-024 확장) 으로 cap monitoring 가능

## Consequences

- 긍정:
  - 운영 attention cost 보호 (일일 promoted 10 max + weekly publication 1)
  - 첫 발행 시간 박싱 강화 (DEC-005 의 2주 목표 lock 보강)
  - "수집 vs triage" 비대칭 명시 → Discovery polling 의 implementation
    우선순위 = score-rank + LLM-judge (not "모든 item 수집")
  - 결정 도입 최소화 원칙 — 새 ADR/DEC 의 churn 방지
- 부정:
  - cap 이 너무 좁아 첫 발행 의외의 source / claim 누락 가능 — 운영자
    명시 override 옵션 (예 `--ignore-cap` flag) 권고
  - "수집 5~10% promote" 비율이 source quality 에 따라 변동 — 일률 적용
    X, 운영 누적 후 source 별 priority weight 조정
  - 결정 도입 최소화 원칙은 enforcement 약함 (운영자 self-discipline)
- Follow-ups:
  - INFRA-1B.2 (Discovery worker) — candidate cap 20 + score-rank 알고리즘
  - EXTR-1A.4 (Review queue) — promoted cap 5~10 + 알람
  - AGG-1A.1 (Dossier composer) — weekly cap 1 + 운영자 명시 trigger
  - REQ-024 / ADR-0020 metrics_run 확장 — 4 신규 metric
  - 첫 발행 retrospective 작성 의무 (`docs/10_PROJECT_RETROSPECTIVE.md`
    갱신) — v0 turn-key 발행 후
