# DEC-006: Vault sync trigger — git push가 자체 사이트 build trigger (Q-026 resolve)

## Context

Q-026은 vault sync trigger를 (a) publication 시점 자동 / (b) manual only /
(c) hybrid 중에서 hybrid(publication 자동 + 나머지 manual)로 권고했다.

ADR-0022 결정으로 자체 사이트의 source가 vault publications/ 디렉토리 그대로
(`glob('vault/publications/**/*.{md,mdx}')`)가 됐다. 외부 sync 명령이 필요
없어졌으며 vault sync trigger 정책도 단순화 가능.

## Decision

vault sync trigger를 **git commit + push 단일화**로 lock한다:

1. **자체 사이트 build trigger**: vault repo에 push 발생 시 Cloudflare Pages
   가 자동 build + deploy. publications/ 디렉토리 변경분이 사이트에 반영.
2. **vault publications/ 외 디렉토리**(dossiers / scenarios / theses /
   content_drafts / promoted_claims) 는 자체 사이트에 노출되지 않음
   (ADR-0022 INV-0022-4). 이 디렉토리 변경은 **사이트 산출물에 영향
   없음** (publications/ 만 source). 단 **build trigger 자체는 Cloudflare
   Pages Build watch paths 가 명시 설정된 경우에만 publications/ 외 변경에서
   발생 안 함** (아래 Precondition 섹션 참조) — watch paths 미설정 시 모든
   push 가 build 를 trigger 해 free tier 월 500 build 카운트를 소모.
   따라서 PUB-1A.4 진입 시점에 watch paths 설정은 의무.
3. **별도 `pipeline vault-sync` 명령은 v0에서 불필요**. v1+에서 외부 플랫폼
   (Substack / YouTube / X) auto cross-post 도입 시 (Q-033) 별도 sync 명령
   재검토.
4. **Publication state 변경** (live → corrected → retracted) 도 동일 trigger.
   ContentDraft frontmatter `status` + `correction_ledger[]` 수정 → git push
   → Cloudflare Pages build → `<RetractionBanner/>` + `<CorrectionLedger/>`
   컴포넌트가 visible 렌더.

## Precondition — Cloudflare Pages Build watch paths

Cloudflare Pages는 기본적으로 repo 내 모든 파일 변경에 build를 trigger한다.
INV-0022-4의 "publications/ 외 디렉토리는 사이트 비노출" 의 의미가 build-trigger
관점에서도 성립하려면 **Build watch paths를 명시 설정**해야 한다. 미설정 시
사이트 산출물은 변하지 않더라도 build 자체는 매 push에 실행 — Cloudflare
Pages free tier(월 500 build) 카운트 소모.

권고 watch paths:

```yaml
include:
  - site/**                  # Astro 사이트 코드 + layout + 컴포넌트
  - vault/publications/**    # ADR-0022 INV-0022-1 single source
  - package.json
  - astro.config.*           # mjs / ts
  - src/content.config.*     # Astro Content Collection schema (Zod)
exclude:
  - vault/dossiers/**        # internal (ADR-0022 INV-0022-4)
  - vault/scenarios/**       # internal
  - vault/theses/**          # internal
  - vault/content_drafts/**  # drafts (publications/로 promote 전)
  - vault/promoted_claims/** # internal cite anchor target
  - vault/documents/**       # internal hub
  - docs/**                  # project docs (ADR/DEC/Q/PRD 등) — 사이트 비노출
```

정확한 디렉토리명은 실제 repo 구조 + Astro 프로젝트 구성에 맞춰 조정.
PUB-1A.4 진입 시점에 Cloudflare Pages 프로젝트 설정 + `docs/05_RUNBOOK.md`
의 "Publishing Site Deployment" 섹션에 commit.

## Rationale

- ADR-0022 가 vault publications/ 를 single source로 lock하면서 "publication
  시점 자동 sync"의 의미가 git push 자체로 흡수됨 — 별도 sync 워크플로 불필요
- Cloudflare Pages 의 git integration 이 default 워크플로 — 운영자 추가 명령
  없음
- "manual sync" 옵션 (Q-026 의 (b)) 은 정정 cascade 의 visibility 지연을
  유발 — corrected/retracted state 가 즉시 사이트에 반영되어야 cite check
  5+1 정합 보존
- "hybrid" (Q-026 의 (c)) 의 "publication 자동 + 나머지 manual" 분기도 ADR-
  0022 후로는 의미 없음 — publications/ 가 single source이고 외 디렉토리는
  자체 사이트에 노출되지 않으므로 manual sync 명령이 필요한 객체가 없음

## Consequences

- 긍정:
  - sync trigger 단순화 (git push 단일) — 별도 CLI 명령 / cron 미필요
  - Publication state 변경 즉시 사이트 반영 → cite check 5+1 의 retracted
    state visibility 보존 (ADR-0022 INV-0022-3, INV-0022-7)
  - vault 외 디렉토리 변경은 build trigger 발생해도 publications/ 변경이
    없으면 사이트 변경 없음 — Cloudflare Pages incremental
- 부정:
  - v1+에서 외부 플랫폼 auto cross-post 도입 시 (Q-033) 별도 sync 워크플로
    재도입 필요 — 본 DEC는 v0 한정
  - 잘못된 push (예 status 오타) 가 사이트에 즉시 반영 → Astro Zod schema
    가 build fail로 차단 (ADR-0022 INV-0022-3), 그러나 build fail 자체는
    visible (Cloudflare Pages 의 이전 deploy 유지) — 운영자가 build log
    모니터링 필요
- Follow-ups:
  - Slice `PUB-1A.3` scope 갱신 — "vault sync trigger" 를 git push 단일화로
    명시 (cron / 별도 CLI 미포함)
  - Slice `PUB-1A.4` — Cloudflare Pages 프로젝트 설정 시 Build watch paths
    위 precondition대로 commit. `docs/05_RUNBOOK.md` "Publishing Site
    Deployment" 섹션에 동일 watch paths + build failure / rollback workflow
    명시
  - HLD `pipeline vault-sync` CLI 명령은 v0에서 제거 (또는 v1+ deferred로
    표시)
  - Q-033 (외부 플랫폼 auto cross-post timing) 결정 시 본 DEC 재검토 — auto
    cross-post 도입 시 별도 sync 명령 재도입 가능성
