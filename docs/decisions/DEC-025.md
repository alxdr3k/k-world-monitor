# DEC-025: Article extraction strict JSON schema v1 — minimum field set lock

## Context

Cycle 42 follow-up #2 (PR #102) 가 `ArticleExtractor` 에 DEC-010 §8 strict
schema wiring 을 도입했다. 도입 시점에 schema 의 field set / strictness /
vendor 추상화 형식 / extended field 의 v1 포함 여부가 코드에만 존재하고
canonical decision 으로 기록되지 않아, 다음 두 risk 가 발생:

1. **drift risk** — 후속 PR 이 schema 에 임의로 field 를 추가/제거할 때
   기준이 코드 diff 뿐. canonical decision 없이 코드가 schema source-of-
   truth 가 되면 ADR-0023 (LLM routing) + AC-010 (c) 와의 cross-ref 가
   끊김.
2. **ordering risk** — schema parse 가 run_ledger missing-cost guard
   보다 먼저 실행되면, vendor 가 malformed JSON 과 `totalCostUsd` 누락
   을 동시에 반환할 때 missing-cost guard 가 우회되어 billable API call
   이 AC-019 daily aggregation 에서 누락. 코드 도입 시점에는 이 ordering
   이 문서화되어 있지 않았음.

본 DEC 는 PR #102 의 schema 결정 본문을 canonical 로 lock 하고, ordering
invariant 를 명시적으로 정의한다.

## Decision

### D1 — Article extraction v1 schema (minimum field set)

**Field set** (3 root field, claim-level 2 field, 모두 required):

| Path | Type | Required | 설명 |
|---|---|---|---|
| `title` | string | yes | 추출된 기사 제목. 원문 title 의 paraphrase 또는 verbatim 허용. |
| `summary` | string | yes | 1~3 문장 factual summary. opinion / 평가 금지. |
| `key_claims` | array (≥ 0) | yes | 추출된 claim 의 ordered list. empty array 허용 (no claims = valid extraction). |
| `key_claims[i].claim` | string | yes | claim 본문 (paraphrase 허용). |
| `key_claims[i].evidence_quote` | string | yes | 원문 sanitized body 의 EXACT substring (verbatim quote). |

**JSON Schema 형식 (OpenAI strict json_schema 기준)**:

```json
{
  "type": "json_schema",
  "json_schema": {
    "name": "article_extraction",
    "strict": true,
    "schema": {
      "type": "object",
      "additionalProperties": false,
      "required": ["title", "summary", "key_claims"],
      "properties": {
        "title": { "type": "string" },
        "summary": { "type": "string" },
        "key_claims": {
          "type": "array",
          "items": {
            "type": "object",
            "additionalProperties": false,
            "required": ["claim", "evidence_quote"],
            "properties": {
              "claim": { "type": "string" },
              "evidence_quote": { "type": "string" }
            }
          }
        }
      }
    }
  }
}
```

**Vendor 추상화 (DEC-010 §8 mapping)**:

| Vendor | wire 형식 |
|---|---|
| OpenAI | `response_format: { type: "json_schema", json_schema: { name, strict: true, schema } }` 그대로 |
| Anthropic | tool input schema + `strict: true` (Anthropic Claude tool-call strict mode) |
| Google | `responseSchema` (Gemini) |
| Mock client (테스트) | parameter pass-through 만 검증 |

**TypeScript surface naming**: snake_case (wire) → camelCase (코드).
`key_claims` → `keyClaims`, `evidence_quote` → `evidenceQuote`. 변환
single-source 는 `parseExtractedArticle()`.

**Strictness invariant**:
- root + claim-level 모두 `additionalProperties: false`
- 모든 정의된 field 는 `required` (OpenAI strict 모드의 의무 — strict
  mode 에서는 optional field 사용 시 client-side 거부)
- empty `key_claims` array 는 허용 (no claims = valid extraction)

### D2 — Run ledger ordering invariant (PR #102 codex round 1 P2)

`ArticleExtractor.extract()` 의 LLM invoke 이후 control flow 의무 순서:

```
1. invoke (on throw: failRun + rethrow)
2. missing-cost guard
   - if (ledgerEnabled && llmResult.totalCostUsd === undefined)
       → failRun(runId) + throw "totalCostUsd missing"
3. schema parse (parseExtractedArticle)
   - on throw: failRun(runId, {inputTokens, outputTokens, cachedTokens?, totalCostUsd, modelId?}) + rethrow
4. completeRun (or failRun on completeRun validation throw)
```

**Invariant**: missing-cost guard (단계 2) 가 schema parse (단계 3) 보다
**먼저 실행** 되어야 한다.

**Rationale**: vendor 가 malformed JSON 과 `totalCostUsd` 누락 을 동시에
반환하는 경우, parse-first 순서 에서는 schema-fail 경로 가 먼저 trip
되고, 그 경로 의 `failRun` payload 는 `totalCostUsd` undefined-spread 에
의해 누락된 채로 row 를 닫음. 그 후 throw 가 발생하므로 missing-cost
guard 가 **영원히** 실행되지 않고 row 의 `total_cost_usd = NULL` 상태로
billable API call 이 AC-019 daily aggregation 에서 사라짐. missing-cost
guard 의 존재 이유 자체가 "vendor 가 cost 를 잘못 보고하는 시점에 그
fact 를 surface 하는 것" 이므로, schema parse 보다 우선되어야 한다.

### D3 — Extended schema field 의 v1 미포함 (deferred)

다음 field 는 v1 schema 에 **포함하지 않는다**. 후속 DEC 또는 ADR
없이 추가 금지:

- `entities` (예: org / person / location list)
- `dates` (event date / publication date 추출 분리)
- `perspective_polarity` / `stance` / `claim_direction`
- `confidence_score` / `importance_score`
- `topic_tags` / `subtopic_tags`
- `relations` (entity-relation triple)

Re-entry gate:
- claim promotion + semantic dedup ADR (Q-057 / DEC-024 D6 의 ADR-NNNN
  defer 의 산물) 작성 후, 해당 ADR 가 어느 field 가 schema-level extraction
  vs post-hoc enrichment 인지 결정
- 또는 첫 발행 retrospective (PUB-1A.5) 의 실측 결과 가 minimum schema
  로는 부족 함 을 evidence-bear 한 경우

extended field 를 ad-hoc 으로 추가하는 PR 은 본 DEC 의 lock 위반.

### D4 — Quote substring 검증 의 v1 미포함 (deferred)

DEC-010 §8 의 "post-LLM quote substring 검증 의무 (faithfulness_rate ≥
0.99 KPI)" 는 v1 schema wiring 에 포함하지 **않는다**.

근거:
- schema wiring (json_schema parameter 전달 + parse) 와 faithfulness
  validation (sanitized body 안에서 `evidence_quote` substring 일치
  검사) 은 분리된 concern
- substring 검증은 separate slice (가칭 `EXTR-1A.2c` 또는 후속 cycle
  follow-up) 의 책임
- 본 DEC v1 schema 는 schema-level 형식 + run_ledger ordering 까지만
  lock; semantic-level faithfulness 는 별도 진입

Re-entry gate: AC-010 (c) 의 "post-LLM quote substring 검증" 항목을
구체 implementation 으로 진입할 때 별도 DEC 또는 본 DEC 갱신 시점에
faithfulness validator interface + threshold 정의.

## Rationale

- minimum field set 으로 시작 — DEC-010 §8 의 strict schema 의무 를
  만족하는 가장 작은 surface. 운영자 2026-05-19 답변 "minimal 로 하고
  나머지 extended 는 별도 슬라이스로" 를 canonical 로 기록.
- empty `key_claims` 허용 — "claim 추출 실패 = LLM 호출 실패" 가 아닌
  "본문에 추출할 claim 이 없음 = valid factual statement" 시나리오
  (예: pure event report) 를 schema-level 에서 차단하지 않기 위함.
- additionalProperties false (root + claim) — OpenAI strict mode 의무
  + vendor 가 임의 field 추가 시 strict mode parse fail 으로 즉시
  surface (silent drift 차단).
- ordering invariant (D2) — codex round 1 P2 finding 의 직접 결과.
  ordering 자체를 코드 comment 만으로 두면 다음 refactor 가 schema parse
  를 어디든 옮길 수 있음. DEC 로 lock 하여 cross-doc 안전성 확보.
- D3 의 extended field 명시 deferral — Q-057 / DEC-024 D6 의 claim
  promotion ADR 미작성 상태에서 schema 에 perspective_polarity /
  confidence_score 등을 추가하면 ADR 작성 시점에 다시 schema-rewrite
  필요. v1 minimum 으로 진입 → ADR 작성 → 그 ADR 의 결정에 따라 v2
  schema (별도 DEC) 진입 의 ordering.

## Consequences

- 긍정:
  - article extraction schema 의 v1 surface 가 canonical 로 lock —
    다음 agent 가 코드만 보고 schema 결정을 reverse-engineer 할 필요
    없음
  - PR #102 의 ordering invariant 가 코드 comment 외에 DEC 로도 명시 —
    schema parse 위치를 임의로 재배열하는 refactor 가 즉시 본 DEC §D2
    위반으로 탐지
  - extended field 추가 PR 이 본 DEC 의 lock 위반 으로 명시 차단 —
    "schema 에 entities 추가" 류 ad-hoc PR 이 review 단계 에서 reject
    가능 (ADR-NNNN 작성 + 본 DEC 갱신 또는 후속 DEC 이전)
- 부정:
  - extended field 가 필요한 downstream consumer (예: scenario validator
    가 perspective_polarity 를 schema-level 에서 받기를 원함) 는 v1
    schema 만으로는 부족 — claim promotion ADR 작성 까지 post-hoc
    enrichment 로 대체
  - empty `key_claims` 가 허용되므로 "LLM 이 추출 자체를 포기" vs
    "본문에 claim 없음" 은 schema-level 구분 불가 — 후속 validator
    slice 에서 별도 heuristic 필요 (예: summary length / claim 0 +
    body length 큰 경우 warn)
  - quote substring 검증 (D4) 가 deferred 이므로 v1 단계에서는
    faithfulness_rate KPI 측정 불가 — AC-010 (c) KPI 는 substring
    validator slice 진입 시점부터 측정 가능
- Follow-ups:
  - Slice `EXTR-1A.2c` (가칭) — D4 quote substring validator 진입.
    sanitized body + evidenceQuote 의 EXACT substring 일치 검사 + KPI
    측정 wiring
  - ADR-NNNN (Q-057 / DEC-024 D6 의 claim promotion + semantic dedup +
    claim_direction) — D3 의 extended field re-entry gate
  - PUB-1A.5 첫 발행 retrospective — D3 extended field re-entry 의
    또 다른 trigger (실측 evidence 기반)
  - PR #102 본 PR 안에서 D2 ordering fix (parse-before-cost-check → cost-check-before-parse) 동시 적용
