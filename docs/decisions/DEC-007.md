# DEC-007: Retention / R2 lifecycle / backup schedule lock — Q-027 resolve

## Context

Q-027은 백업 schedule + R2 lifecycle을 제안. research file
`docs/research/retention-policy-2026-05.md` (2026-05-11) 가 hybrid model
(R2 native lifecycle for age-based expiry + application batch jobs for
semantic GC) 을 권고. 사용자가 "권고대로 lock" 결정.

## Decision

research 권고 9개 항목 + Q-A~Q-G micro 결정 7개를 lock한다.

### Core 9 lock items

1. **R2 lifecycle expire rule은 단 3개만**:
   - `backups/neo4j/` — expire 30d
   - `backups/sqlite/` — expire 90d
   - `tmp/multipart/` — abort incomplete 7d (R2 default)
2. **R2 transition rule (5개) — expire 절대 없음**:
   - `audit/jsonl/` — IA transition 365d
   - `permitted_artifact/dataset/` — IA transition 30d (EOL시 batch job이 archive prefix로 copy 후 delete)
   - `permitted_artifact/derived/snapshot/` — IA transition 365d
   - `permitted_artifact/derived/dossier/` — IA transition 365d
   - `permitted_artifact/derived/publication/` — IA transition 730d (expire 절대 없음 — 항상 보존)
   - `permitted_artifact/evidence-pack/` — IA transition 365d
3. **의미적 GC는 application batch job 3개**:
   - 일간 `retention_daily`: raw_cache_items TTL evict, research_session GC (30d after closed_at), tmp computation purge
   - 주간 `retention_weekly`: snapshot supersede collapse (1y grace + cited 보호), content_hash dedupe canonical, scenario/thesis revision collapse (head + recent 5)
   - 월간 `retention_monthly`: dataset_vintage EOL processing (12mo grace), metrics rollup, JSONL audit export, publication 정정 cascade integrity check, tombstone GC report
4. **모든 delete는 2단계 패턴**: soft-delete → tombstone(`:Tombstone` 노드 + reason + 14d grace) → hard-delete. 2단계 사이에 `policy_decisions` row 1개씩 발행
5. **RETENTION_PROTECTED_KINDS 상수**: GC batch가 reject해야 할 항목 list — Publication 노드 + derived/publication/{id}/ R2 객체, AccessIntervention ledger 전체, policy_decisions, JSONL audit export, Claim retracted tombstone, Publication에 cited된 Snapshot(cited flag), dataset_vintage row metadata, Tombstone 노드 전체
6. **raw_cache_items**: R2 진입 절대 금지(ADR-0012). local FS + SQLite metadata only. TTL 24h~7d (ADR-0021 INV-0021-6). ttl_override는 24h~7d clamp. fetched_at hard ceiling + last_accessed_at LRU 둘 다 같은 clamped effective_ttl 사용. finalize/abandon 시 동기 즉시 삭제
7. **IA transition은 1년 이상 cold-ish data만**. retrieval cost 미미 — retroactive verification에 부담 없음
8. **Q-027 권장값 lock**: Neo4j dump 30d + SQLite 90d + JSONL audit 1y hot → 무기한 IA + derived publication 무기한 + dataset versioned + EOL grace 12mo
9. **연 1회 retention drill**: 임의 publication 1건 골라 5+1 trace 전부 reachable 한지 수동 verify. fail → batch 정책 재검토

### Q-A~Q-G micro decisions (research 권고 채택)

- **Q-A** (Snapshot cited lock 시점): 한 번 cited되면 영구 lock. publication 자체가 retracted돼도 unlock 안 함 (retraction 자체가 evidence — audit lineage 보존)
- **Q-B** (metrics rollup 정밀도): raw row 12mo + **monthly aggregated rollup** (daily aggregation은 raw 12mo 안에서만, 월단위로 rollup → 무기한 aggregate)
- **Q-C** (JSONL audit IA transition 시점): **1년 후 IA, 무기한 보존**
- **Q-D** (dataset_vintage EOL 후 archive prefix 운명): **무기한 IA**. 5년 후 delete 안 함 — 5-10년 horizon 시나리오에서 vintage 재현 가능성 보존
- **Q-E** (raw_cache_items TTL default): **`ttl_ceiling` = 7d default**. 운영자가 명시 `ttl_override` 지정 시 24h~7d 범위로 clamp. ttl_floor(24h) 는 LRU idle window의 lower clamp일 뿐 default 아님
- **Q-F** (tombstone retention): **무기한 보존**. 10년 보더리 millions row 도달 시 aggregate rollup 별도 Q로 분리 (현 시점 미해결, 별도 Q 신설 필요 시 v2+)
- **Q-G** (R2 object versioning): **OFF**. 1인 단일 작성자 운영에서 versioning은 lifecycle 충돌 + storage 비용 추가. application batch tombstone이 안전망 역할

## Rationale

- R2 native lifecycle은 age + prefix만 — 의미적 기준(cited 여부, retraction, supersede, dedupe, EOL)을 모름. 따라서 native + batch hybrid가 자연
- expire를 1~2개로 최소화하면 "실수로 publication까지 삭제됨" 사고가 구조적으로 불가능
- soft-delete + tombstone 패턴은 모든 retention 결정의 reversibility 14d 보장 — 1인 운영자가 retention drill에서 실수 발견 시 grace 안에서 복구 가능
- raw_cache_items LRU window 정합성 (codex P2 fix 반영) — `ttl_floor`를 LRU에 쓰면 운영자 명시 `ttl_override` 무력화
- 비용: 1인 운영자 보수적 상한에서 1년 R2 비용 USD 8~20 (egress=0). free tier 10GB로 1차년도 backup 대부분 흡수

## Consequences

- 긍정:
  - 모든 retention policy가 코드와 ledger로 enforce 가능 (3개 R2 rule + 3개 batch + RETENTION_PROTECTED_KINDS 상수)
  - 5+1 trace retroactive verification이 항상 가능 (cited Snapshot 영구 lock)
  - audit lineage 보존 (Tombstone 무기한)
  - 비용 USD 0.65~1.50/month (1년 USD 8~20) — 1인 운영 부담 미미
- 부정:
  - Tombstone 무기한 보존이 10년 boundary에서 millions row 도달 가능 — 그 시점에 aggregate rollup 별도 Q 필요 (현 시점 미해결)
  - dataset_vintage IA grace 12mo + 무기한 IA는 storage 비용 누적 (보수적 100GB 가정 USD 1.10/month)
  - R2 versioning OFF는 application batch tombstone 의존 — batch bug 시 안전망 부재. 단 14d grace 안에서 복구 가능
- Follow-ups:
  - Slice `INFRA-1A.8` (백업 runbook): cron schedule + R2 lifecycle JSON 정책 + RETENTION_PROTECTED_KINDS 상수 commit
  - Slice `OPS-1A.4` (raw_cache_items TTL worker): research file Section 5.1 의사코드 그대로 구현 (LRU window = clamped effective_ttl)
  - 연 1회 retention drill을 `docs/05_RUNBOOK.md` 에 추가
  - 별도 Q (v2+ 시점): Tombstone aggregate rollup 정책 (10년 boundary millions row 대비)
