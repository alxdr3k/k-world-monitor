# DEC-008: LLM cost / quality discipline lock — Q-028 resolve

## Context

Q-028은 LLM API cost 통제 정책 (prompt caching + batch + per-day spend
ceiling) 을 제안. research file `docs/research/llm-cost-quality-2026-05.md`
(2026-05-11) 가 3계층 라우팅 + prompt caching layering + batch API +
cost ceiling + 강제 tool-use schema + quote substring 검증 + KPI 5개를
권고. 사용자가 "권고대로 lock" 결정.

## Decision

research 권고 9개 lock items + 사용자 결정 필요 항목 A~F 권고 채택을
모두 lock한다.

### Core 9 lock items

1. **모델 라우팅 3계층**:
   - Haiku 4.5 1차 (extract, dossier, ContentDraft, EvidencePack, cite check LLM-judge)
   - Sonnet 4.6 escalate (confidence < 0.85 OR reliability_tier ∈ {medium,low} OR domain ∈ specialty(감염병/거시)). Scenario composer 기본도 Sonnet
   - Opus 4.7 선택적: Scenario validate (3') + high-stakes Thesis composer (주 1회 빈도)
2. **Prompt caching layering**: tools → system → glossary → source_policy → few-shot → [breakpoint] → dynamic. 5분 TTL 기본
3. **이중 TTL**: batch 호출은 5분 TTL, sync 호출은 **1시간 TTL** (2× write cost 수용)
4. **Batch API 기본 ON**: extract(article/report), dossier 합성, ContentDraft 4-format, EvidencePack. Sync: Scenario composer/validate, Thesis composer, cite check 발행 게이트
5. **강제 tool-use + strict schema**: 모든 LLM 호출에 `tool_choice` + `strict: true` 적용. `anthropic-beta: structured-outputs-2025-11-13` header
6. **Quote substring 검증**: post-LLM deterministic 검증 — `quote_reason`이 원문 substring 아니면 결과 폐기 + retry. faithfulness_rate ≥ 0.99 게이트
7. **Temperature**: 0 (extract / cite check / Thesis), 0.3 (Scenario branch 다양성)
8. **Cost ceiling**:
   - soft throttle $5/day (큐 backoff + 알람)
   - hard stop $7.5/day (신규 LLM 호출 차단, 진행 중만 완료)
   - weekly $25
   - **backfill은 별도 budget bucket** (예 backfill_run_id 태그, 평소 ceiling과 분리 집계)
9. **KPI 5개 + run_ledger 필드 확장**:
   - cost_per_promoted_claim ≤ $0.005/claim
   - faithfulness_rate ≥ 0.99
   - auto_confirm_fp_rate ≤ 0.05 (SPIKE-002)
   - cache_hit_rate ≥ 0.70 (SPIKE-003, 실제 85%+ 기대)
   - counterclaim_coverage ≥ 0.90
   - run_ledger 필드: prompt_version + system_prompt_sha256 + cached_tokens + batch_id

### 사용자 결정 항목 A~F (research 권고 채택)

- **A** (Opus 4.7 적용 범위): scenario validate + high-stakes thesis만. 다른 곳은 ROI 음수
- **B** (Specialty 도메인 list): v0 = 감염병 + 거시. 추가는 운영 중 누적 (별도 Q 신설 가능)
- **C** (Cache TTL 정책): 이중 (sync 1시간 + batch 5분)
- **D** (Backfill budget bucket): 별도 bucket. backfill_run_id 태그로 평소 ceiling과 분리 집계
- **E** (LLM-judge cite check 도입 단계): v0 deterministic 룰만, v1+ LLM-judge "warn only" 단계로 추가
- **F** (SPIKE-003 cache hit 측정 시 batch 5분 초과 분모): cache miss로 카운트하되 별도 라벨(`tll_expired_miss`)로 구분 — hit rate 정의 명확화

### 비용 시뮬레이션 (boundary)

- 보정 baseline ≈ $1.50~$1.70/day (cache/batch 적용 전)
- cache 90% + batch 50% stacking 후 실효 ≈ $0.40~$0.60/day
- 5배 여유는 (a) 대량 backfill (b) counterclaim 적극 탐색 (c) batch retry 폭주 (d) specialty 도메인 도배 시 무너질 수 있음 — soft $5/hard $7.5는 합리적

## Rationale

- 모델 선택보다 prompt 설계가 효과 큼 (Haiku→Sonnet 정확도 5~15% 개선 vs strict schema JSON validity 80~95% → 99.5%+)
- Quote substring 검증 1개가 hallucinated quote 위험을 거의 0으로 — 모델 크기 무관
- Self-critique는 high-accuracy task에서 corrosive → extract 1차 미적용, counterclaim/scenario validate에만 적용
- Opus 4.7은 GDPval-AA SOTA + source-grounded 21% 적은 오류 — scenario validate 발행 직전 1회는 정당화. 다른 곳은 token efficiency 우위만으로는 비용 정당화 어려움
- Cache TTL 이중화: batch는 1시간 안에 들어오므로 5분 TTL이면 첫 호출만 write, 나머지 read. sync는 1시간 TTL이 cache miss 줄임

## Consequences

- 긍정:
  - 비용 baseline $0.40~$0.60/day 달성 가능 (cache 90% + batch 50% stacking)
  - faithfulness_rate ≥ 0.99 (quote substring 검증 + strict schema)
  - run_ledger에 prompt_version + system_prompt_sha256 + cache 추적 → cache invalidation reproducibility
  - 3계층 라우팅으로 model spend 최적화 (Opus는 주 1~2회, Sonnet escalate ~10%, Haiku 80%+)
- 부정:
  - Cache TTL 이중화 코드 복잡도 ↑ (batch vs sync 분기)
  - Opus 4.7 적용 결정 룰이 운영 데이터 누적 후에야 정합 (SPIKE-002 + Opus ROI 검증 필요)
  - Specialty 도메인 list가 v0 2개만 — 추가 도메인 도입 시 별도 Q
  - Backfill budget bucket은 분리 집계 필요 — run_ledger schema에 `backfill_run_id` flag 추가 필요
- Follow-ups:
  - Slice `EXTR-1A.2` (Article extractor Haiku 4.5): tool_choice + strict schema + quote substring 검증
  - Slice `EXTR-1A.3` (Sonnet escalate + prompt caching + batch API): 이중 TTL + batch ON
  - Slice `OPS-1A.1` (run ledger): prompt_version + system_prompt_sha256 + cached_tokens + batch_id + backfill_run_id 필드
  - Slice `OPS-1A.2` (cost throttling): soft $5 / hard $7.5 / weekly $25 + backfill bucket
  - Slice `OPS-1A.3` (metrics framework v0): KPI 5개 측정
  - SPIKE-002 (auto_confirm threshold 0.85): EXTR-1A.4 진입 전 검증
  - SPIKE-003 (cache hit rate ≥ 70%): EXTR-1A.3 진입 전 검증
  - 별도 Spike 후보 #A/#B (Sonnet escalate trigger rule + Opus ROI): 운영 데이터 200건 누적 후 정식 SPIKE ID 발급
  - 별도 Q (Specialty 도메인 추가 — Q-031~034 외 신설 시점): 운영 중 누적 시
