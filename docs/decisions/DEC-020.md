### [Q-042] — NFR-003 5-hop trace 의무 reflow (10-stage 도입 이후)

채택: **옵션 (b) + ADR-0011 의 원본 기준 의미 명시 lock**.

**5-hop 의 원래 기준 조사 결과** (`docs/adr/0011-pipeline-object-model-9-stage-source-thesis.md:203-205`):

> 5-step trace(Publication → ContentDraft → Thesis → Scenario → Claim →
> Snapshot → R2 bytes)는 9-stage 안에서 5단계 이내 유지 가능 (선택적 단계
> skip, NFR-003)

trace path 는 9-stage 전체가 아니라 **6 노드 / 5 edge** (Publication →
ContentDraft → Thesis → Scenario → Claim → Snapshot). Source layer 의 4-tier
(Source / Document / Snapshot / Claim) 중 Source / Document 는 trace 종점이
아니라 Snapshot 이 종점. R2 bytes 는 ADR-0012 raw_cloud_policy=always_
prohibited 로 supersede 됐으므로 종점은 Snapshot (fingerprint).

**ADR-0025 INV-0025-5 가 이미 EditorialIntent 추가 시 의도 명시**:

> ContentDraft + Publication 의 cite check 5+1 trace path 에 EditorialIntent
> 가 포함 — Publication → ContentDraft → Thesis → **EditorialIntent** →
> Scenario revision → Dossier → promoted Claim → Snapshot → Source. 10-stage
> 안에서도 NFR-003 의 5단계 trace 는 선택적 단계 skip 으로 유지 (예
> ContentDraft → Thesis → EditorialIntent → Scenario revision → Claim →
> Snapshot 6 hops 도 가능하나 5단계 단축 path 보존).

즉 ADR-0025 본문에 "5단계 단축 path 보존" 의무가 이미 lock 됐다. 본 Q 는
그 의무를 NFR-003 본문 표현으로 surface — silent drift closure.

**reflow 후 NFR-003 의미**:

- **단축 trace path (5-hop, NFR-003 의무)**: Publication → ContentDraft
  → Thesis → Scenario → Claim → Snapshot. 9-stage / 10-stage 양쪽에서 동일.
- **EditorialIntent (ADR-0025 신규 stage)**: Thesis 의 anchor metadata 로
  분류. trace 계산 시 선택적 skip 허용 (INV-0025-2 Thesis ↔ EditorialIntent
  1:1 link 으로 reachable 보장됨 — trace depth 계산에서 한 단계 압축).
- **Source layer 의 Source / Document**: trace 종점은 Snapshot. Source /
  Document 는 Snapshot 의 metadata anchor (publisher / URL group / reliability
  _tier) — trace depth 계산에서 종점 이후 단계가 아닌 종점 위의 직접
  attribute.
- **manual_claim_entry path (ADR-0018)**: Snapshot 단계 미포함 — 4-hop
  으로 추가 단축 (Publication → ContentDraft → Thesis → Scenario → Claim).
- **측정**: cite check report 의 trace depth.

### [Q-045] — invariant validator scope 확장 (ADR INV ↔ 코드 cross-ref)

채택: **옵션 (b) — frontmatter `cross_ref_code[]` 필드 + validator 검증
로직 추가 (기본 권고)**.

- ADR / DEC frontmatter schema 확장:
  ```yaml
  invariants:
    - id: INV-0012-3
      statement: ...
      cross_ref_code:
        - "src/storage/r2/policy.ts:checkPermittedPrefix"
        - "src/discovery/worker/snapshot-fingerprint.ts:allLinkedSourcesAllowRawCloud"
  ```
- 형식: **file:path:exportName** (안정성 우선) + 옵션으로 `file:line` (precise
  하지만 fragile). 두 형식 모두 허용 — validator 가 각각의 존재 검증.
- validator (`scripts/validate_invariants.ts`) 안에 cross_ref_code 검증
  로직 추가 (~50 줄):
  - file:exportName: grep export 또는 AST scan
  - file:line: line 존재 + 해당 line context 가 comment / signature 인지
    1-line check
  - 깨진 reference 는 warning (ADR-0002 INV-0002-1 warning-level by contract
    준수)
- boilerplate sync 정책: 본 repo local extension 으로 먼저 lock. 작동 검증
  후 boilerplate PR 로 promote 검토 (별도 PR).
- 우선 backfill: INV-0012-3 (raw_cloud_policy) / INV-0028-1~6 (safe-fetch
  8단 방어) / INV-0023-3 (domain_override_reason 의무) / INV-0017 audit.
- 구현 슬라이스: 신규 슬라이스 `INFRA-1A.9-validator-extension` 등록.

**Semantic scope 명시 (Opus PR #66~#78 adversarial review F3 wording correction, GPT 합의 — 2026-05-18)**:
`cross_ref_code` entry 는 **enforcement proof 가 아니라 reachability
heuristic** 이다. 운영자 / 후속 reviewer 가 entry 존재만으로 "이 invariant
의 enforcement 가 코드에서 닫혀 있다" 고 읽으면 안 된다. 구체적 semantic
weight 는 entry 형식에 따라 다르다:

- `file:exportName` = **`export_exists` 수준 weak evidence**.
  validator 가 확인하는 것은 해당 파일이 `export <kw> <exportName>` 선언
  또는 `export { <exportName> }` re-export block 을 가지고 있다는 것뿐.
  function body 가 invariant statement 의 실제 enforcement 를 수행하는지는
  확인하지 않는다. body 가 비어 있어도 export 만 있으면 ok 처리.

- `file:line` = **`line_exists` 수준 더 weak evidence**.
  validator 가 확인하는 것은 line number 가 1..lineCount 범위 안이라는
  것뿐. 코드 삽입 / 삭제로 해당 line 이 다른 함수 / 주석 / blank 로
  밀려도 validator 는 계속 green 을 반환한다. private helper 참조에만
  사용 — 가능한 경우 named export 로 promote 후 `file:exportName` 형식
  사용 권장.

따라서 `cross_ref_code` 는 "이 invariant 와 관련된 코드 site 가 존재
한다" 는 reachability anchor 이지, "이 invariant 가 enforcement 로
닫혀 있다" 는 proof 가 아니다. ADR / PR body 에서 `cross_ref_code` 추가를
"enforcement proven" / "invariant closed" 류 표현으로 쓰면 안 된다. 정확한
표현은 "reachability anchor landed" / "invariant 의 코드 site cross-ref
backfill landed" 등.

`callsite_exists` (function call 의 존재) / `forbidden_call_absence`
(금지 호출의 부재) / `test_fixture_proved` (test 가 invariant 를
실제로 verify) 같은 stronger semantic level 은 본 validator scope 외 —
별도 follow-up anchor 로 register 가능 (INFRA-1A.9.h2-cross-ref-semantic-level
후보, NOT gate-blocking). 본 DEC-020 Q-045 는 (b) reachability heuristic
level 만 채택.
